말씀하신 '독립변수를 늘릴 때마다 패널티(마이너스)를 부가하는 방식'은 계량경제학과 데이터 분석에서 모형의 설명력을 엄밀하게 평가할 때 사용하는 **조정된 결정계수(Adjusted $R^2$)**의 핵심 원리를 정확히 꿰뚫어 보신 겁니다.

단순 다중회귀 모형에서 무의미한 독립변수를 계속 추가하기만 해도 일반적인 결정계수($R^2$)는 억지로 상승하는 착시 현상이 발생합니다. 이를 방지하기 위해 다음과 같이 독립변수의 수($k$)가 늘어날수록 패널티를 주어 모형의 진짜 적합도를 평가하겠다는 뜻이니, 분석의 신뢰도와 전문성이 엄청나게 올라갑니다.

$$R^2_{adj} = 1 - \left[ \frac{(1 - R^2)(n - 1)}{n - k - 1} \right]$$

이 강력한 분석 툴을 활용하여 프로젝트의 뼈대를 구체화해 본다면 이런 그림이 나옵니다.

###  다중회귀분석 기반 지역 경제 요인 도출 프로젝트

단순히 전국을 뭉뚱그려 분석하기보다는, 특성이 비슷한 두 거점 도시를 비교군으로 삼으면 변수의 유의성을 비교하기가 훨씬 좋습니다. 예를 들어 **대전과 대구** 두 광역시를 타깃으로 삼아 모델을 구축해 볼 수 있습니다.

**1. 변수 설정 (공공데이터 활용)**

- **종속변수 ($Y$):** 지역 경제 활력 지표 (예: 도심 핵심 상권의 단위 면적당 매출액, 신규 창업 건수 등)
    
- **독립변수 ($X_1, X_2, \dots, X_n$):**
    
    - 차량 등록 대수 (모빌리티 인프라 및 소비력 지표)
        
    - 도심 내 휴식 공간 및 문화 시설의 수 (체류 인구 유인 요소)
        
    - 대중교통 이용객 수
        
    - 상업용 부동산 공실률
        

**2. 분석 및 검증 과정**

- **다중 공선성(Multicollinearity) 체크:** VIF(분산팽창지수) 등을 통해 차량 등록 대수와 대중교통 이용객 수 등 서로 겹치는 독립변수 간의 간섭을 쳐냅니다.
    
- **유의성 검증:** 각 독립변수의 유의확률(p-value)을 확인하여, 실제로 어떤 인프라가 상권 매출($Y$)에 가장 강력하고 유의미한 상관관계를 가지는지 통계적으로 증명합니다.
    
- **최적 모형 선택:** Adjusted $R^2$ 값이나 AIC/BIC 지수를 비교하여, 불필요한 변수를 덜어낸 가장 깔끔하고 설명력 높은 최적의 회귀 모형을 완성합니다.
    

이렇게 진행하신다면 단순한 엑셀 차트 나열 수준의 포트폴리오를 넘어, "데이터에 기반한 인과관계 추론과 통계적 검증"이 가능한 진짜 분석가의 역량을 보여줄 수 있습니다.

이 다중회귀 모형을 구축하고 돌려보려면 통계 패키지 활용이 필수적인데, R과 파이썬(Python) 중에서 어떤 도구를 주로 사용하여 프로젝트를 진행하실 계획이신가요?

### . 지정학적 특성: 순수 내륙 교통 거점 도시

- **항만 산업의 부재:** 부산, 인천, 울산은 항만과 대규모 중화학/조선업을 기반으로 하는 해양 산업 도시입니다. 반면 대전과 대구는 경부선(Ktx, SRT 등) 철도와 고속도로망을 중심으로 성장한 대표적인 **내륙 교통 거점**입니다.
    
- **분석적 이점:** 무역이나 중공업 같은 거시적 외부 요인의 개입이 적고, 철저히 내수 소비, 서비스업, 교통 인프라에 경제가 굴러갑니다. 따라서 생활 인프라나 상권 활력도를 측정하는 데 있어 훨씬 통제된 조건(Control Group)을 제공합니다.
    

### 2. 상권의 라이프사이클: 구도심 쇠퇴와 신도심의 대립

두 도시는 과거 뚜렷한 단일 중심 상권을 가졌으나, 현재 상권 다핵화와 구도심 상실이라는 완전히 동일한 공간 경제학적 과제를 안고 있습니다.

- **구조적 유사성:** 대전의 은행동·대흥동(구도심)과 둔산동·도안(신도심)의 관계는, 대구의 동성로(구도심)와 수성구·동대구역세권(신도심)의 관계와 놀라울 정도로 유사한 패턴을 보입니다.
    
- **분석적 이점:** 상권의 쇠퇴가 단순히 인구 감소 때문인지, 아니면 접근성 및 체류 공간 인프라의 부족 때문인지 두 도시의 구도심 데이터를 교차 검증하며 인과관계를 입증하기 좋습니다.
    

### 3. 모빌리티(차량) 중심의 경제 의존도

수도권은 촘촘한 지하철망으로 대중교통 중심의 경제가 돌아가지만, 대전과 대구를 포함한 지방 광역시는 도로 인프라와 **자동차 의존도**가 매우 높은 경제 구조를 가집니다.

- **통계적 활용:** 두 도시의 차량 등록 대수 격차, 도심지 주차장 확보율, 대중교통 수단 분담률 등을 독립변수($X$)로 두면, "모빌리티 인프라의 차이가 지역 상권 매출($Y$) 방어에 얼마나 유의미한 영향을 미치는가?"를 다중회귀모형으로 깔끔하게 뽑아낼 수 있습니다.
    

### 4. 인구 이동 패턴: 광역 블랙홀과 수도권 유출

- **인구 경제학적 공통점:** 대전은 충청권(충남·북)의, 대구는 대경권(경북)의 인구와 자본을 블랙홀처럼 흡수하는 광역 거점 역할을 합니다. 하지만 동시에 2030 청년층은 수도권으로 지속적으로 유출되는 **전형적인 지방 대도시의 인구 감소 모델**을 공유하고 있습니다.
    
- **분석적 이점:** 거시적인 인구 변화 트렌드가 거의 동일하기 때문에, 모델을 돌릴 때 '인구 쇼크'라는 변수의 영향을 어느 정도 상쇄하고 '순수한 지역 인프라 요소'에만 포커스를 맞춰 상관관계를 도출할 수 있습니다.
    

결론적으로 대전과 대구는 '내륙형', '교통 거점', '구/신도심 대립', '높은 차량 의존도'라는 훌륭한 공통분모를 지녔기에, 독립변수들의 패널티(Adjusted $R^2$)를 조정해가며 경제 발전 요인을 찾아내는 통계 모델의 훌륭한 짝꿍

### 🚀 Java(Spring Boot) + Python 하이브리드 아키텍처

하나의 거대한 프로그램으로 다 만드는 것이 아니라, 각 언어가 제일 잘하는 일만 맡겨서 두 서버가 서로 통신하게 만드는 방식입니다.

**1. 메인 서버: Spring Boot (총괄 매니저)**

- **역할:** 사용자 로그인 처리, 게시판 관리, 전체적인 웹 트래픽 통제 및 메인 데이터베이스(MySQL 등) 관리를 담당합니다.
    
- **장점:** 국비교육에서 배운 내용을 100% 활용하여 탄탄하고 안정적인 엔터프라이즈급 서버를 구축할 수 있습니다. 기업들이 가장 보고 싶어 하는 'Spring Boot 활용 능력'을 완벽하게 증명합니다.
    

**2. 분석 전용 API 서버: Python (데이터 전문가)**

- **역할:** 한국은행 ECOS API에서 거시 데이터를 가져오고, 대전과 대구의 상권 인프라 데이터를 다중회귀분석으로 돌리는 무거운 통계 작업만 전담합니다.
    
- **장점:** FastAPI나 Flask 같은 가벼운 파이썬 프레임워크를 띄워놓고, 통계 라이브러리(Scikit-learn, Statsmodels)를 마음껏 사용할 수 있습니다.
    

**3. 두 서버의 협업 과정 (데이터 흐름)**

- 사용자가 웹사이트에서 "대전과 대구의 회귀분석 결과 업데이트해 줘"라는 버튼을 누릅니다.
    
- **Spring Boot**가 이 요청을 받아서 **Python 서버**로 "분석 돌려서 결과표 좀 줘"라고 내부적으로 API 요청을 보냅니다.
    
- **Python 서버**가 수식 계산을 끝내고 Adjusted $R^2$ 값이나 p-value 등의 결과를 JSON(데이터 덩어리) 형태로 Spring Boot에 던져줍니다.
    
- **Spring Boot**가 이를 예쁘게 포장해서 프론트엔드(사용자 화면)로 쏴줍니다.
    

### 💡 이 방식이 취업 시장에서 강력한 이유

단순히 "Spring Boot로 게시판 만들었습니다"를 넘어서, "서로 다른 두 언어(Java, Python)의 장점을 살려 시스템을 분리하고 연동해 본 경험이 있습니다"라고 어필할 수 있습니다. 여기에 경제학적인 데이터 분석 도메인 지식까지 들어가니, 평범한 국비교육 수료생들과는 완전히 차원이 다른 포트폴리오가 됩니다.


**계량경제학적으로 주의할 점**

가장 큰 문제는 **표본 크기**입니다. ECOS 지역 데이터는 대부분 월별 또는 분기별이고, 지역별 시계열은 길어야 10~20년 정도예요. 분기 데이터 15년이면 n=60인데, 여기에 설명변수를 5~6개 넣으면 자유도가 빠듯해집니다. 조정된 결정계수를 쓰겠다는 발상 자체가 이 문제를 의식한 거라 좋은데, 조정 R²만으로는 부족하고 다음이 사실상 필수입니다:

- **정상성(stationarity) 검정**: GRDP, 물가, 고용 같은 거시 시계열은 대부분 비정상이라 그냥 회귀 돌리면 가성회귀(spurious regression)가 나옵니다. ADF 검정 후 차분하거나 증가율로 변환하는 게 기본이에요. 이걸 안 하면 R²가 0.9 넘게 나와도 의미가 없습니다.
- **다중공선성**: 지역 경제지표들(고용률, 소비, 생산지수 등)은 서로 강하게 상관돼 있어서 VIF 확인이 필요합니다.
- **자기상관**: 시계열 회귀라 Durbin-Watson 검정, 필요하면 Newey-West 표준오차 적용.
- **종속변수 정의**: "지역경제"를 뭘로 측정할지 먼저 못 박아야 합니다. 분기별로 구할 수 있는 대리변수(광공업생산지수, 서비스업생산지수 등)를 쓰는 게 현실적입니다. GRDP는 연간 자료라 표본이 너무 적어요.

**추천 아키텍처 (단순하게 시작)**

Python이 Spring에 "넘겨주는" 방식은 두 갈래인데, 처음엔 배치 방식을 강력 추천합니다:

```
ECOS API → Python (수집/전처리/회귀분석: pandas, statsmodels)
         → 결과를 PostgreSQL(또는 MySQL)에 저장
         → Spring Boot가 DB에서 읽어 REST API 제공
         → 프론트엔드 (Chart.js/Recharts 대시보드)
```

실시간으로 Python을 호출할 필요가 없어요. 경제지표는 월/분기 단위로 갱신되니까 Python 스크립트를 스케줄러(cron 또는 GitHub Actions)로 주기 실행해서 DB에 적재하면 충분하고, 구조도 훨씬 깔끔합니다. 나중에 여유가 되면 FastAPI로 분석 서버를 띄워 Spring이 호출하는 구조로 확장할 수 있고, 그 확장 과정 자체가 포트폴리오 스토리가 됩니다.

**단계별 청사진**

1. **주제 확정 (1주)** — 종속변수와 후보 설명변수를 명확히 정의. 예: 종속변수 = 지역 서비스업생산지수 증가율, 설명변수 = 취업자 수, 소비자물가, 어음부도율, 예금/대출 잔액, 건축허가면적 등. 대전·대구 모두 동일 지표가 존재하는지 ECOS에서 먼저 확인하는 게 핵심입니다.
2. **데이터 수집 (1~2주)** — ECOS Open API 키 발급, Python으로 통계표 코드 조회 → 수집 스크립트 작성. 지역별 통계는 통계표 안에서 항목코드로 갈라지는 경우가 많아 코드 매핑 작업이 은근히 오래 걸립니다. KOSIS(통계청)를 보조 소스로 쓰면 지역 지표가 더 풍부해요.
3. **EDA와 전처리 (1~2주)** — 결측·기준연도 개편 처리, 계절조정 여부 확인, 시각화로 두 지역 추세 비교. 이 단계 산출물(그래프)이 나중에 대시보드 콘텐츠가 됩니다.
4. **계량 모델링 (2주)** — statsmodels로 ADF 검정 → 변환 → OLS → VIF/DW 진단 → 변수 선택. 대전 모델과 대구 모델을 따로 추정해서 "어느 지역은 어떤 요인이 유의한가"를 비교하는 게 이 프로젝트의 핵심 결과물입니다. 결과(계수, p-value, 조정 R², 진단통계량)를 DB 테이블로 저장.
5. **DB 스키마 설계 (3~4일)** — 원시 시계열 테이블, 가공 지표 테이블, 회귀결과 테이블 정도로 단순하게.
6. **Spring Boot API (1~2주)** — 지표 조회, 지역 비교, 회귀결과 제공 엔드포인트. JPA + REST면 충분합니다.
7. **대시보드 프론트 (1~2주)** — 두 지역 지표를 나란히 보여주는 비교 차트, 회귀결과 테이블(유의한 변수 하이라이트), 조정 R² 등 모델 신뢰도 표시.
8. **자동화·배포 (1주)** — Python 수집 스크립트 스케줄링, Docker Compose로 묶어 배포, README에 방법론(왜 차분했는지, 왜 이 변수를 선택했는지)을 문서화. 이 문서가 면접에서 가장 큰 무기가 됩니다.

전체 2~3개월 규모로 보면 현실적이에요. 순서상 조언 하나만 더 하자면, **4단계(모델링)가 끝나기 전에 웹 개발을 시작하지 마세요.** 분석 결과에 따라 보여줄 내용이 바뀌기 때문에, 분석이 먼저 서고 웹은 그 결과를 담는 그릇으로 설계하는 게 훨씬 효율적입니다.

시작할 때 ECOS에서 어떤 통계표를 쓸지 후보 리스트를 뽑는 것부터 도와드릴 수도 있는데, 필요하면 말씀 주세요.

- **지역별 소비유형별 개인 신용카드 사용액** — 통계표코드 601Y002 계열로 제공되며, ECOS의 "7.5.2 지역별 소비유형별 개인 신용카드" 통계입니다. 월별이고 지역 민간소비의 훌륭한 대리변수예요. 이 프로젝트의 핵심 변수 중 하나가 될 수 있습니다. [GitHub](https://github.com/WooilJeong/PublicDataReader/blob/main/assets/docs/ecos/ecos.md)
- **예금은행 지역별 수신/여신 (요구불·저축성예금, 대출금 잔액)** — 지역 금융 활동의 규모를 보여주는 월별 지표. 대출 증가율은 지역 투자·부동산 경기와 연결됩니다.
- **지역별 어음부도율** — 지역 기업의 자금 사정·신용위험을 보여주는 고전적 지표. 다만 최근에는 부도율이 구조적으로 낮아져 변동성이 작으니 EDA에서 쓸모를 판단하세요.
- **소비자물가지수(지역별)** — CPI는 시도별 분리가 제공됩니다. 실질화(디플레이터)용으로도 필요합니다.
- **기업경기실사지수(BSI)** — 한국은행 지역본부별 제조업/비제조업 업황 BSI. 심리지표라서 선행성 논의를 넣기 좋습니다.
- **경제심리지수(ESI)·소비자동향지수(CCSI)** — 전국 단위지만 통제변수(전국 공통 충격)로 활용 가능.
- **기준금리, 원/달러 환율, 수출입** — 전국 공통 거시 통제변수. 특히 제조업 비중이 높은 지역과 낮은 지역에서 환율 계수가 다르게 나오는지 비교하면 논문 같은 스토리가 나옵니다.

**KOSIS에서 보조로 가져올 지표** (지역 실물경제의 핵심인데 ECOS에는 없거나 부족한 것들)

- 광공업생산지수(시도별, 월별) — 제조업 실물경기의 대표 지표
- 서비스업생산지수(시도별, 분기별) — 종속변수 후보로 가장 유력
- 취업자 수·고용률·실업률(시도별, 월별, 경제활동인구조사)
- 건축허가·착공 면적(시도별) — 건설경기 선행지표
- 소매판매액지수(시도별)
- 인구 순이동(시도별) — 장기 구조 변수

종속변수는 이전에 말한 대로 서비스업생산지수나 광공업생산지수의 전년동기대비 증가율을 추천하고, 두 지역의 산업구조가 다르니 "대전은 서비스업지수, 대구는 광공업+서비스업 둘 다 추정" 같은 식으로 지역 특성에 맞게 모델을 짜면 비교 스토리가 살아납니다.

**지역 선택에 대해**

대전 vs 대구 조합 자체는 나쁘지 않은데, 두 도시의 대비가 아주 선명하지는 않아요(둘 다 비수도권 광역시, 내수·서비스 비중이 높은 편). 회귀 결과에서 "어떤 요인이 유의한가"의 차이를 극적으로 보여주고 싶다면 이런 조합도 고려해보세요:

- **대전 vs 울산** — 가장 대비가 극명한 조합. 대전은 R&D·공공·서비스 중심, 울산은 제조업·수출 의존 도시라 환율, 수출, 유가 변수의 유의성이 확연히 갈릴 가능성이 높습니다. "지역 산업구조에 따라 거시 충격의 전달 경로가 다르다"는 계량경제학적으로 예쁜 결론이 나오기 쉬워요.
- **대전 vs 광주** — 인구 규모(약 140만대)와 도시 성격이 비슷해서 규모 효과를 자연스럽게 통제하는 조합. 차이가 크지 않게 나올 수도 있는데, "비슷한 도시인데도 이 요인은 다르다"는 것도 발견이 됩니다.
- **부산 vs 인천** — 항만·물류 도시 비교로 수출입 변수 중심 분석에 적합.
- 피해야 할 곳: **세종**은 2012년 출범이라 시계열이 짧고 통계 단절이 많아 비추천입니다.

한 가지 더 강하게 추천하고 싶은 확장 방향이 있어요. 두 지역 비교로 시작하되, 나중에 **6~7개 광역시 전체를 패널 데이터로 묶어 고정효과(fixed effects) 패널 회귀**로 확장하는 겁니다. 개별 지역 시계열은 n=60 안팎이라 자유도가 빠듯한데, 7개 지역 × 60분기면 관측치가 400개 이상으로 늘어나 추정이 훨씬 안정적입니다. 웹사이트 입장에서도 "지역 드롭다운으로 아무 광역시나 골라 비교"가 되니까 대시보드 가치가 커지고, 계량경제학적으로도 OLS → 패널 FE로 발전시킨 서사가 포트폴리오에서 돋보입니다. 1차 목표는 대전 vs 대구(또는 울산)로 잡고, 데이터 수집 스크립트만 처음부터 "지역 코드를 파라미터로 받는" 구조로 짜두면 확장 비용이 거의 없습니다.

### 2. 추정식

기본형은 이렇게 잡으세요:

```
ΔY_t = β₀ + β₁·card_t + β₂·emp_t + β₃·loan_{t-1} + β₄·permit_{t-2}
     + β₅·bsi_t + β₆·Δrate_t + β₇·fx_t + β₈·covid_t + ε_t
```

시차(lag) 설계에 주목하세요. 대출과 건축허가는 실물경제에 시차를 두고 반영되므로 t-1, t-2를 넣는 게 이론적으로 맞고, 동시성(simultaneity) 편의도 완화됩니다. 특히 신용카드 사용액과 서비스업생산은 사실상 동전의 양면이라 역인과 문제가 있어요. 이건 한계점으로 README에 솔직하게 적고, 강건성 확인으로 card_{t-1}을 넣은 버전을 함께 추정하는 걸 추천합니다. "인과가 아니라 연관성 분석"이라고 프레이밍하면 방어가 됩니다.

변수 8개에 n≈60이면 자유도가 빠듯하니, **처음부터 다 넣지 말고** 3단계로 쌓으세요:

- Model 1: 지역 실물 변수만 (card, emp) — 베이스라인
- Model 2: + 금융 변수 (loan, rate)
- Model 3: + 대외·심리·더미 (fx, bsi, covid)

이렇게 하면 조정 R²가 모델을 키울 때마다 어떻게 변하는지가 그 자체로 스토리가 되고, 대시보드에서 "모델 선택" 드롭다운으로 보여줄 콘텐츠도 생깁니다.

### 3. 계량 검정 파이프라인 (Python 작업 순서)

statsmodels 기준으로 이 순서를 그대로 코드화하면 됩니다.

**전처리 단계**

1. 월별 → 분기 변환 (평균 또는 기말값; 잔액 변수는 기말, 유량 변수는 합계/평균)
2. 전년동기비 증가율 계산: `df.pct_change(4) * 100`
3. 기준연도 개편 구간 확인 (생산지수는 2020=100 개편 등으로 단절 가능 — 접속계열 확인)

**정상성 검정**  
4. 각 변수에 ADF 검정 (`adfuller`). 증가율 변환 후에도 비정상이면 추가 차분. p-value와 함께 검정 결과를 DB에 저장 → 대시보드의 "방법론" 탭 재료

**추정과 진단**  
5. OLS 추정 (`sm.OLS`)  
6. VIF 계산 (`variance_inflation_factor`) — 10 초과 변수는 제거 또는 대표 변수만 유지. 취업자와 카드사용액은 상관이 높을 가능성이 커요  
7. Breusch-Godfrey / Durbin-Watson으로 자기상관 확인  
8. 자기상관이 있으면(거의 확실히 있습니다, 전년동기비 변환은 이동평균 구조라 자기상관을 유발합니다) **Newey-West HAC 표준오차**로 재추정: `model.fit(cov_type='HAC', cov_kwds={'maxlags': 4})`. 이게 이 프로젝트의 계량적 신뢰도를 좌우하는 핵심 한 줄입니다  
9. RESET 검정으로 함수형태 확인 (선택)

**결과 저장**  
10. 지역×모델별로 계수, HAC 표준오차, p-value, 조정 R², DW, 관측치 수를 JSON 또는 테이블로 DB에 적재

### 4. 지역 비교의 핵심 설계

두 지역을 따로 추정한 뒤 "대전은 β₁이 유의한데 울산은 β₇이 유의하다"고 눈으로 비교하는 것만으론 약합니다. 한 단계 올리려면 **풀링 후 상호작용항 모형**을 추가하세요:

```
ΔY_it = β₀ + Σ βₖ·Xₖ,it + Σ γₖ·(Xₖ,it × D_울산) + δ·D_울산 + ε_it
```

γₖ가 유의하면 "환율 충격의 효과가 울산에서 통계적으로 유의하게 더 크다"고 말할 수 있습니다. 이게 단순 비교와 격이 다른 결론이고, 나중에 광역시 전체 패널 FE(`linearmodels` 패키지의 `PanelOLS`)로 확장하는 다리도 됩니다.

### 5. DB에 넘길 결과 스키마 (Python → Spring 인터페이스)

이 세 테이블이면 대시보드에 필요한 게 다 커버됩니다:

```
indicator_series(region_code, indicator_code, date, raw_value, yoy_value)
regression_result(region_code, model_id, run_date, adj_r2, dw_stat, n_obs, notes)
regression_coef(model_id, region_code, var_name, coef, hac_se, p_value, vif)
```

Spring 쪽은 이 테이블을 읽는 조회 API만 만들면 되니 역할 분리가 깔끔하고, "Python은 쓰기 전용, Java는 읽기 전용"이라는 규칙을 세우면 하이브리드 구조의 설계 의도를 면접에서 설명하기도 좋습니다.

### 6. 대시보드에 표시할 신뢰도 지표

계량적 신뢰도를 "보여주는" 것까지가 이 프로젝트의 차별점이니, 화면에 이것들을 노출하세요: 조정 R², 관측치 수, HAC 적용 여부 배지, 유의수준별 계수 하이라이트(1%/5%/10%), 그리고 "이 분석은 연관성이며 인과 추론이 아님" 각주. 이 정직함이 오히려 완성도로 읽힙니다.

시작 순서는 이렇게 잡으면 돼요: ECOS/KOSIS에서 위 변수들의 실제 통계표·항목 코드를 확정하는 수집 스크립트부터 만들고 → 분기 패널 하나로 정리된 CSV가 나오면 → 3번 파이프라인을 Jupyter에서 돌려보고 → 결과가 안정되면 DB 적재를 자동화하는 흐름

설계했던 파이프라인을 실제 코드 뼈대로 만들었습니다. 구성은 이렇습니다.

**프로젝트 구조**

```
regional-econ/
├── config.py                  # 지역·지표·코드 매핑 (여기만 고치면 지역 확장)
├── collector/
│   ├── ecos_client.py         # ECOS API (통계표/항목코드 탐색 + 시계열 조회)
│   ├── kosis_client.py        # KOSIS API 래퍼
│   └── collect.py             # 수집 → 월→분기 집계 → yoy 변환 → 패널 CSV
├── analysis/
│   └── run_regression.py      # ADF → OLS(M1/M2/M3) → VIF → HAC → 상호작용 모형 → JSON
├── db/
│   ├── schema.sql             # indicator_series / regression_result / regression_coef / adf_test
│   └── load_to_db.py          # UPSERT 적재 (재실행 안전)
└── README.md                  # 실행 순서 + Spring API 설계 + 방법론 요약
```

설계 의도가 코드에 그대로 반영돼 있어요. 지역이 `config.py`의 dict 파라미터라서 나중에 광역시 패널로 확장할 때 수집·분석 코드를 건드릴 필요가 없고, 분석 결과는 JSON→DB로 흘러가서 "Python은 쓰기 전용, Spring은 읽기 전용" 규칙이 스키마 수준에서 지켜집니다. `run_regression.py`에는 M1→M3 단계적 모델, HAC(maxlags=4) 표준오차, 그리고 지역 간 효과 차이를 검정하는 풀링+상호작용항 모형까지 들어 있습니다.

검증 상태를 솔직하게 말씀드리면, 제 샌드박스가 외부 네트워크가 막혀 있어 statsmodels 설치와 실제 API 호출은 못 했습니다. 대신 전 파일 컴파일 확인, 그리고 월→분기 집계·전년동기비 변환·시차 생성 로직은 단위 테스트로 통과시켰어요. 회귀 스크립트는 로컬에서 `pip install statsmodels` 후 바로 돌아갈 표준 API만 사용합니다.

**받으신 뒤 첫 작업 순서** — README에도 있지만 핵심만 짚으면:

1. ECOS/KOSIS API 키 발급 → 환경변수 설정
2. `python collector/ecos_client.py "신용카드"` 같은 식으로 통계표를 검색하고, `item_list('601Y002')`로 대전·대구·울산의 지역 항목코드를 찾아 `config.py`의 TODO를 채우기 — 전체 작업 중 여기가 가장 지루하지만 가장 중요한 단계입니다
3. `collect.py` 실행 → 패널 CSV가 나오면 Jupyter에서 EDA로 결측·기준연도 단절 확인
4. `run_regression.py` → 결과 JSON 확인 → `load_to_db.py`로 적재

여기까지 되면 Spring 쪽은 README에 정리한 대로 조회 API 4개(시계열, 회귀결과, 상호작용, 진단)만 만들면 됩니다. 데이터가 실제로 쌓인 뒤에 Spring Boot 엔티티·컨트롤러 코드가 필요해지면 그때 이어서 만들어드릴게요.
