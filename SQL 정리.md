# **1과목 데이터 모델링의 이해 - 1장 데이터 모델링의 이해**

## **1. 데이터 모델의 이해**

### **1.1 모델링의 정의**

**모델링(Modeling)**: 현실 세계의 복잡한 현상이나 사물을 단순화하여 표현하는 기법

- 데이터진흥원 정의: 사람이 접하는 다양한 현상(사람, 사물, 개념 등)을 약속된 방식(수식, 그림 등)으로 표현하는 것

### **1.2 모델링의 종류**

- 정보시스템 모델링: 정보 모델링, 데이터 모델링, 프로세스 모델링 등
- 수리 모델링: 수학적 공식으로 현상을 표현
- 확률/통계 모델링: 통계적 방법론으로 미래를 예측
- 회로 모델링: 전자회로를 설계하고 구성

### **1.3 모델링의 특징(**<font color="#0070c0">단.추.명</font>**)

- **단순화**: 복잡한 현실을 기호, 문자, 그림 등으로 쉽게 이해할 수 있게 함
- **추상화**: 현실 세계를 일정한 형식에 맞추어 간략하게 표
- **명확화**: 애매모호함을 제거하고 명확하게 현상을 기술

### **1.4 정보시스템 모델링의 세 가지 관점**

- 데이터 관점 (What, Data): 업무가 어떤 데이터와 관련되는지 모델링
- 프로세스 관점 (How, Process): 업무가 실제로 처리하는 일을 모델링
- 데이터와 프로세스의 상관 관점: 업무 처리 방식에 따라 데이터가 어떤 영향을 주고받는지 모델링

### **1.5 데이터 모델링의 정의**

데이터 모델링: 정보시스템 구축을 위해 데이터 관점에서 업무를 분석하고 현실 세계의 데이터를 약속된 표기법으로 표현하는 과정

데이터 모델이 제공하는 기능:

- 시스템 가시화 지원
- 시스템 구조와 행동 명세화
- 구조화된 틀 제공
- 문서화
- 다양한 관점 제공
- 상세 수준의 구체화

### **1.6 데이터 모델링의 중요성 및 유의점**

- 데이터 설계의 중요성:
- 시스템 구축에서 매우 중요한 요소
- 복잡한 정보 요구사항을 간결하게 표현
- 데이터 중복/비유연성/비일관성을 줄여 품질 향상
- 데이터 모델링 시 유의점:
- 중복된 데이터 요소 최소화
- 변화 가능성 고려 (업무 변화에 유연하게 대응)
- 데이터 모델의 단순화 추구
- 데이터 재사용성 고려
- 일관성과 통합성 유지

### **1.7 데이터 모델링의 3단계**

#### **개념적 데이터 모델링**: Keyword : 추상화, 전사적, ERD

- 추상화 수준이 높고 업무 중심적, 포괄적 수준의 모델링
- 전사적 관점에서 기업의 데이터 모델링 진행(전 부서가 공통으로 사용할 수 있는 통합된 데이터 모델)
- 주로 **ERD** 산출물 생성

#### **논리적 데이터 모델링**:

- 시스템 구축을 위한 키, 속성, 관계 등을 정확하게 표현
- 재사용성이 높은 설계가 가능
- 정규화를 통해 데이터 모델 품질 향상

#### **물리적 데이터 모델링**:

- 실제 DB 구축을 위한 설계 단계
- 성능, 저장 등을 고려한 실제적 구현 방법 제시
- 인덱스, **역정규화(=반정규화)** 등의 기법 사용
- **역정규화:** 정규화된 모델을 **다시(역) 합치거나 중복을 허용**하여 성능을 높이는 작업

### **1.8 데이터 독립성 (Three-Schema Architecture)**

**외부 스키마 (External Schema):**

- 사용자 관점의 스키마
- 개인적인 데이터베이스 뷰
- 응용 프로그램이 접근하는 영역

**개념 스키마 (Conceptual Schema):**

- 통합 관점의 스키마
- 조직 전체의 데이터베이스를 기술
- 전체 데이터베이스의 구조 정의

**내부 스키마 (Internal Schema):**

- 물리적 저장구조
- 실제 데이터가 저장되는 방법 정의

**독립성 종류:**

- 논리적 독립성: 개념 스키마가 변경되어도 외부 스키마에 영향 없음

쉽게 말해, **DB 전체 구조를 조금 수정해도, 개별 사용자가 쓰는 화면(View)에는 아무런 문제가 없어야 한다**

- 물리적 독립성: 내부 스키마가 변경되어도 개념 스키마에 영향 없음

저장 장치를 바꾸거나 파일을 다르게 저장해도, **DB의 전체적인 구조(ERD 등)는 그대로 유지**되어야 한다

- 매핑(사상): 상위 단계와 하위 단계 사이의 대응 관계
    
- **논리적 독립성 = 사용자 뷰를 보호**
    
- **물리적 독립성 = 물리적 저장 구조를 보호**
    

### **1.9 데이터 모델링의 세 가지 핵심 요소**

- 엔터티 (Entity): 업무에 필요하고 유용한 정보를 저장하는 집합적 실체
- 속성 (Attribute): 엔터티를 구성하는 최소 데이터 단위
- 관계 (Relationship): 엔터티 간의 논리적인 연관성

### **1.10 ERD (Entity-Relationship Diagram)**

ERD 정의: 엔터티와 엔터티 간의 관계를 그림으로 표현한 것

ERD 작성 순서:

1. 엔터티 도출
2. 엔터티 간 관계 정의
3. 관계명 기술
4. 관계의 참여도 기술
5. 관계 필수여부 기술

ERD 표기법 종류:

- Chen 표기법: 가장 오래된 표기법, 관계를 마름모꼴로 표시
- **IE (Information Engineering) 표기법**: 까마귀발 표기법이라고도 함, 관계를 선으로 표시
- **Barker 표기법**: 엔터티에 주식별자를 표기하지 않음, 관계는 선으로 표시

### **1.11 좋은 데이터 모델링의 특징**

- 완전성: 업무에 필요한 모든 데이터를 담아야 함
- 중복 배제: 데이터 중복을 최소화하여 일관성 유지
- 업무 규칙 반영: 현업의 업무 규칙을 정확히 표현
- 데이터 재사용: 데이터 재사용이 용이하도록 설계
- 의사소통: 비즈니스 요구사항을 명확히 전달
- 통합성: 전체 데이터의 통합된 시각을 제공

## **2. 관계형 데이터베이스 개요**

### **2.1 데이터베이스의 정의**

데이터의 집합으로 조직적으로 구성된 데이터들의 모임

여러 관점의 정의:

- 법적 정의: 저작물의 소재를 체계적으로 배열 또는 구성한 편집물
- 기술적 정의: 관련된 데이터의 모음으로, 중복을 통제하는 체계
- 산업적 정의: 조직체의 정보 요구를 지원하는 공유 데이터 자원

### **2.2 데이터베이스의 발전 과정**

- 1960년대: 파일 시스템 (파일 간 데이터 중복, 일관성/무결성 보장 어려움)
- 1970년대: 계층형/망형 DBMS (자료 간 관계가 복잡, 유연성 부족)
- 1980년대: 관계형 DBMS 등장 (오라클, DB2 등)
- 1990년대: 객체 관계형 DBMS로 발전

### **2.3 관계형 데이터베이스의 특징**

- 데이터 중복 최소화: 정규화를 통한 중복 제거
- 동시성 관리: 다중 사용자 환경에서 데이터 일관성 보장
- 병행제어: 동시에 여러 사용자가 데이터 처리할 때 일관성 유지
- 메타데이터 관리: 데이터에 관한 데이터(데이터 사전) 관리
- 보안/무결성/장애복구 기능: 데이터 보호와 안정성 보장
- SQL 사용: 표준 질의어를 통한 데이터 정의/조작/제어

### **2.4 SQL (Structured Query Language)**

SQL의 정의: 관계형 데이터베이스에서 사용하는 표준 질의어

SQL 표준화: ANSI SQL을 통한 표준화

SQL 명령어 유형:

- DML (Data Manipulation Language): SELECT, INSERT, UPDATE, DELETE
- DDL (Data Definition Language): CREATE, ALTER, DROP, TRUNCATE
- DCL (Data Control Language): GRANT, REVOKE
- TCL (Transaction Control Language): COMMIT, ROLLBACK, SAVEPOINT

### **2.5 테이블 구조와 데이터 타입**

테이블 구성 요소:

- 컬럼 (Column): 속성, 필드
- 행 (Row): 레코드, 튜플
- 필드 (Field): 컬럼과 행이 만나는 지점, 실제 데이터가 저장됨

주요 데이터 타입:

- 숫자형: INTEGER, NUMBER, DECIMAL 등
- 문자형: CHAR, VARCHAR, TEXT 등
- 날짜형: DATE, TIMESTAMP
- 이진형: BLOB, BINARY
- 불리언형: BOOLEAN
- CHAR vs VARCHAR: CHAR은 고정길이, VARCHAR는 가변길이 문자열 저장

## **3. 엔터티**

### **3.1 엔터티의 정의**

엔터티: 업무에 필요하고 유용한 정보를 저장하고 관리하기 위한 집합적인 실체 또는 객체

- 데이터베이스에서는 테이블로 구현됨

### **3.2 엔터티의 특징**

- 업무적 필요성: 반드시 업무에서 필요하고 관리할 가치가 있어야 함
- 식별자: 유일한 식별자(예: 학번, 사번)로 식별 가능해야 함
- 인스턴스 집합: 2개 이상의 인스턴스(행)를 포함해야 함
- 업무 프로세스 참여: 실제 업무 프로세스에 활용되어야 함
- 속성 포함: 반드시 속성을 가져야 함
- 관계 존재: 다른 엔터티와 최소 1개 이상의 관계가 필요

### **3.3 엔터티의 분류**

유무형에 따른 분류:

- **유형 엔터티**: 물리적 형태가 있음 (예: 학생, 사원, 강사)
- **개념 엔터티**: 개념적인 정보 (예: 조직, 보험상품)
- **사건 엔터티**: 업무 수행에 따라 발생 (예: 주문, 청구)

발생시점에 따른 분류:

- 기본 엔터티: 독립적으로 생성 가능 (예: 고객, 상품)
- 중심 엔터티: 기본 엔터티로부터 생성 (예: 계약, 주문)
- 행위 엔터티: 두 엔터티 간의 행위로 생성 (예: 주문목록, 사원변경이력)

### **3.4 엔터티 명명 규칙**

- 현업업무에서 사용하는 용어 사용
- 약어 사용 지양
- 단수 명사 사용
- 모든 엔터티에서 유일한 이름 부여

## **4. 속성**

### **4.1 속성의 정의**

속성: 업무에서 필요로 하는 최소한의 데이터 단위로, 더 이상 분리되지 않는 정보 **원자**

- 엔터티를 설명하는 데이터 요소

### **4.2 속성의 특성**

- 업무 필요성: 해당 업무에서 필요하고 관리해야 할 정보여야 함
- 함수적 종속성: 정규화 이론에 따라 주식별자에 함수적 종속성을 가져야 함
- 단일값 원칙: 하나의 속성은 한 개의 값만을 가짐
- 도메인: 속성이 가질 수 있는 값의 범위를 정의

### **4.3 속성의 분류**

특성에 따른 분류:

- 기본 속성: 업무 분석 단계에서 바로 정의되는 속성 (예: 상품이름)
- 설계 속성: 업무 규칙화, DB 설계 과정에서 도출되는 속성 (예: 예금분류코드)
- 파생 속성: 다른 속성에서 계산이나 변형으로 생성되는 속성 (예: 이자)

엔터티 내 역할에 따른 분류:

- **PK(Primary Key) 속성**: 엔터티를 식별하는 속성
- FK(Foreign Key) 속성: 다른 엔터티와의 관계에서 포함된 속성
- 일반 속성: PK나 FK가 아닌 일반적인 속성

### **4.4 속성의 명명 규칙**

- 현업에서 사용하는 이름 사용
- 약어 사용 지양
- 전체 시스템에서 유일한 이름 사용
- 명사형 사용
- 이름만으로도 의미 파악 가능하게

## **5. 관계**

### **5.1 관계의 정의**

관계: 엔터티의 인스턴스 간에 논리적인 연관성을 가지는 형태나 행위

엔터티 간의 업무적 연관성을 표현

### **5.2 관계의 표현**

- 2개의 엔터티 사이의 관계는 2개의 관계명을 가짐
- 예: 고객은 주문을 생성한다 / 주문은 고객에 의해 생성된다

### **5.3 관계의 분류**

종류:

- 존재에 의한 관계: 항상 존재하는 관계 (예: 사원은 부서에 속한다)
- 행위에 의한 관계: 특정 행위로 발생하는 관계 (예: 고객이 상품을 구매)

카디널리티(참여자 수):

- 1:1 관계: 양쪽 엔터티의 인스턴스가 1:1로 대응
- 1:M 관계: 한쪽 엔터티 인스턴스가 여러 인스턴스와 관계
- N:M 관계: 양쪽 엔터티의 인스턴스가 다대다 관계

### **5.4 관계의 선택성**

- 필수적 관계: 반드시 관계를 가져야 함 (실선으로 표시)
- 선택적 관계: 관계를 가질 수도 있고 아닐 수도 있음 (점선으로 표시)

### **5.5 관계명 표기 규칙**

- 관계의 이름은 현재형 동사 사용
- 관계 참여자의 관점에서 능동적으로 명명
- 관계는 양방향으로 명명 (능동/수동)

## **6. 식별자**

### **6.1 식별자의 정의**

식별자: 엔터티 내에서 인스턴스를 구별하는 속성 또는 속성 조합

논리 모델링에서는 '식별자', 물리 모델링에서는 '키(Key)'라는 용어 사용

### **6.2 식별자의 특성**

- 유일성: 인스턴스를 유일하게 구분할 수 있어야 함
- 최소성: 필요 최소한의 속성으로 구성
- 불변성: 값이 변하지 않아야 함
- 존재성: 반드시 값이 존재해야 함 (NULL 허용 안됨)

### **6.3 식별자의 분류**

대표성에 따른 분류:

- 주식별자: 엔터티를 대표하는 식별자 (예: 사원번호)
- 보조식별자: 대체 식별 가능한 식별자 (예: 주민등록번호)

생성 방식에 따른 분류:

- 내부식별자: 엔터티 내에서 스스로 생성되는 식별자 (예: 고객번호)
- 외부식별자: 다른 엔터티로부터 받아오는 식별자 (예: 주문의 고객번호)

속성의 수에 따른 분류:

- 단일식별자: 하나의 속성으로 구성된 식별자 (예: 고객번호)
- 복합식별자: 여러 속성의 조합으로 구성된 식별자 (예: 주문번호+상세순번)

대체 가능성에 따른 분류:

- 본질식별자: 업무적으로 자연 발생한 식별자 (예: 주민등록번호)
- 인조식별자: 시스템에서 임의로 부여한 식별자 (예: 일련번호)

### **6.4 식별자 도출 기준**

- 업무에서 자주 사용되는 속성 우선
- 명칭이나 내역과 같은 서술형 속성은 지양
- 속성의 수가 적은 것 우선
- 향후 변경 가능성이 적은 속성 우선

### **6.5 식별자 관계와 비식별자 관계**

식별자 관계: 부모 엔터티의 주식별자가 자식 엔터티의 주식별자에 포함되는 관계

- 실선으로 표시

비식별자 관계: 부모 엔터티의 주식별자가 자식 엔터티의 일반 속성으로 포함되는 관계

- 점선으로 표시

관계 설정 기준:

- 자식 인스턴스가 부모 없이 독립적 존재 가능 -> 비식별자 관계
- 자식 인스턴스의 존재가 부모에 종속적 -> 식별자 관계
- 복합식별자가 너무 많아지면 비식별자 관계 고려
- 객체 재사용 필요시 비식별자 관계 고려
- 이력 관리 엔터티 식별자 관계 고려

### **정리 및 핵심 포인트**

1. 데이터 모델링은 현실 세계의 데이터를 추상화하여 표현하는 기법으로, **개념적/논리적/물리적** 단계로 진행됩니다.
2. 데이터 모델의 세 가지 핵심 구성요소는 **엔터티(실체),** **속성(특성)**, **관계(연관성)**입니다.
3. 관계형 데이터베이스는 SQL을 통해 데이터를 **정의(DDL), 조작(DML), 제어(DCL)**하며, 데이터 중복을 최소화합니다.
4. 엔터티는 업무에 필요한 정보 집합으로, 유형/개념/사건 엔터티로 나뉘고, 반드시 식별자와 속성을 포함해야 합니다.
5. 속성은 엔터티를 설명하는 최소 데이터 단위로, 기본/설계/파생 속성 및 PK/FK/일반 속성으로 구분됩니다.
6. 관계는 엔터티 간 연관성으로, 1:1, 1:M, N:M의 카디널리티와 필수/선택적 참여 특성을 갖습니다.
7. 식별자는 인스턴스를 구분하는 속성으로, 주/보조 식별자, 내부/외부 식별자, 단일/복합 식별자 등으로 분류됩니다.
8. 좋은 데이터 모델은 완전성, 중복 배제, 업무 규칙 반영, 데이터 재사용성, 명확한 의사소통, 통합성을 갖춰야 합니다.

# **1과목 데이터 모델링의 이해 - 2장 데이터 모델과 SQL**

## **1. 정규화**

### **1.1 정규화의 정의**

**정규화(Normalization)**: 데이터의 일관성, 최소한의 데이터 중복, 최대한의 데이터 유연성을 확보하기 위한 데이터 분해 과정

엔터티를 상호 독립적인 관계로 분리하여 이상 현상(Anomaly)을 제거하는 것이 목적

함수적 종속성(Functional Dependency): 데이터 간의 종속 관계를 기반으로 정규화를 수행

### **1.2 이상 현상 (Anomaly)**

- 삽입 이상(Insert Anomaly): 불필요한 데이터를 함께 삽입해야만 원하는 데이터를 추가할 수 있는 현상
- 갱신 이상(Update Anomaly): 중복 데이터 중 일부만 수정하여 데이터 불일치가 발생하는 현상
- 삭제 이상(Delete Anomaly): 필요한 데이터까지 함께 삭제되는 현상

### **1.3 정규화 단계**

- **제1정규형 (1NF)**
- 모든 속성의 값이 **원자값(Atomic Value)**이어야 함
- 반복 그룹이나 다중값 속성을 제거
- 하나의 속성에는 하나의 값만 존재해야 함

<aside>  
📌

- **Before:** `[학생: 김철수, 과목: 자바, 파이썬]` (한 칸에 두 개!)
    
- **After (1NF):**
    
    - `[학생: 김철수, 과목: 자바]`
    - `[학생: 김철수, 과목: 파이썬]` (각 줄마다 값 1개씩!)  
        </aside>
- **제2정규형 (2NF) -** 1NF를 만족함과 동시에
    
- 부분 함수 종속(Partial Dependency)을 제거
    
- 모든 비식별 속성이 주식별자 전체에 완전 함수 종속이어야 함
    
- 복합 식별자의 일부에만 종속되는 속성을 분리
    

<aside>  
📌

- **Before:** PK가 `[학생 + 과목]`인데, `학생 이름`은 `학생` 정보에만 의존함 (과목과는 상관없음).
    
- **After (2NF):**
    
    - `[학생 테이블: 학생ID(PK), 이름]`
    - `[수강 테이블: 학생ID(PK), 과목ID(PK)]` (이름을 따로 떼어냄!) 만약 성적이라면 완전 함수 종속!  
        </aside>
- **제3정규형 (3NF) -** 2NF를 만족하면서(당연히 1NF도)
    
- 이행 함수 종속(Transitive Dependency)을 제거
    
- 비식별 속성 간의 종속 관계를 제거
    
- A → B → C에서 A → C의 간접 종속이 존재하면 분리
    

<aside>  
📌

- **Before:** `[학생ID(PK) -> 부서번호 -> 부서이름]` (학생ID를 알면 부서번호를 알고, 부서번호를 알면 부서이름을 앎)
    
- **After (3NF):**
    
    - `[학생 테이블: 학생ID(PK), 부서번호]`
    - `[부서 테이블: 부서번호(PK), 부서이름]` (부서 정보를 따로 떼어냄!)  
        </aside>
- **BCNF (Boyce-Codd Normal Form)**
    
- 3NF를 만족하면서 결정자(Determinant)가 모두 후보키이어야 함
    
- 후보키가 아닌 속성이 결정자인 경우를 제거
    
- 모든 결정자가 후보키인 상태
    

<aside>  
📌

- **상황:** `[학생 + 과목]`이 PK인데, 어떤 상황에서는 `교수`만으로도 `과목`를 알 수 있음.
    
- **After (BCNF):** `[과목-교수]` 테이블을 별도로 만듦.  
    </aside>
   
   [직원 테이블]
    - 1번 직원 | 영업부 | 02-111 
    - 2번 직원 | 영업부 | 02-111 
    - 3번 직원 | 영업부 | 02-111

   [직원 테이블] (A -> B)
	• 1번 직원 | 영업부 
	• 2번 직원 | 영업부 
	• 3번 직원 | 영업부

   [부서 테이블] (B -> C) 
    • 영업부 | 02-111
   
- **제4정규형 (4NF)**
    
- 다치 종속(Multi-Valued Dependency)을 제거
    
- 하나의 속성이 여러 개의 독립적인 다중값과 관련되는 경우를 분리
    

<aside>  
📌

- **Before:** 이걸 하나의 테이블에 넣으면 모든 경우의 수를 곱해서 적어야 하므로 데이터가 무의미하게 뻥튀기됩니다.
    
    - `[김철수, 자바, 쇼핑몰]`
    - `[김철수, 자바, 메신저]`
    - `[김철수, 파이썬, 쇼핑몰]`
    - `[김철수, 파이썬, 메신저]` (엄청난 중복과 낭비 발생)
- **After (4NF):** 상관없는 다중값을 쪼개서 테이블을 2개로 완전히 분리합니다.
    
    - `[개발자-스킬 테이블: (김철수, 자바), (김철수, 파이썬)]`
    - `[개발자-프로젝트 테이블: (김철수, 쇼핑몰), (김철수, 메신저)]`  
        </aside>

   [수강 테이블] (PK: 학생 + 과목) 
	• 철수 + 데이터베이스 | 김교수
	• 영희 + 데이터베이스 | 김교수 
	• 철수 + 알고리즘 | 박교수

- **제5정규형 (5NF)**
    
- 조인 종속(Join Dependency)을 제거
    
- 분해된 테이블을 다시 조인했을 때 원래의 테이블과 동일해야 함
    
  -> 수학적으로 조인(주로 NATURAL JOIN이나 INNER JOIN)을 돌렸을 때 원래 상태와 100% 똑같이 원상복구  
### **1.4 정규화와 성능**

- 정규화는 조회 성능을 저하시킬 수도, 향상시킬 수도 있음
- 정규화를 하면 조인이 증가하여 조회 성능이 떨어질 수 있음
- 반면, 중복이 제거되어 데이터 일관성 유지 특정 조회 성능 향상
- 입력/수정/삭제 성능은 일반적으로 향상됨 (중복 데이터 감소로 처리량 감소)

## **2. 반정규화**

### 가장 중요한 개념 하나

> **반정규화는 "정규화를 안 하는 것"이 아니라, "정규화된 걸 일부러 다시 무너뜨리는 것"**

즉, 순서가 중요: **정규화 → (문제 발생: 조회 성능 저하) → 반정규화 적용**

정규화를 하면 테이블이 잘게 쪼개지면서 데이터 무결성은 좋아지지만, 조인이 많아져서 **조회 성능**이 떨어질 수 있어 이걸 보완하려고 일부러 중복/통합을 허용하는 게 반정규화.

**핵심 암기**: "반정규화는 최후의 수단이다" — 성능 문제가 생겼다고 바로 반정규화부터 하는 게 아니라, **구조를 바꾸지 않는 방법(뷰, 인덱스)을 먼저 시도**한다는 게 시험 포인트.

#### 헷갈리기 쉬운 것들 구분

**수직분할 vs 수평분할** (자주 출제)

- 수직분할 = **컬럼**을 나눔 (자주 쓰는 컬럼 / 안 쓰는 컬럼 분리)
- 수평분할 = **행**을 나눔 (지역별, 연도별 등)
- 암기법: "수직은 세로선 = 컬럼을 가로지름", "수평은 가로선 = 행을 가로지름"

**중복 컬럼 추가 vs 파생 컬럼 추가**

- 중복 컬럼: **다른 테이블에 이미 있는 값**을 그대로 복사 (조인 줄이려고)
- 파생 컬럼: **계산해서 나온 값**을 저장 (합계, 평균 등 매번 계산하기 아까우니까)

### 핵심 트레이드오프 (서술형/개념형 문제 대비)

|구분|정규화|반정규화|
|---|---|---|
|목적|데이터 무결성, 중복 제거|조회 성능 향상|
|단점|조인 증가로 성능 저하 가능|데이터 중복으로 **무결성 깨질 위험**|
|특징|입력/수정/삭제(DML)에 유리|조회(SELECT)에 유리|

> **반정규화의 대가**: 데이터가 중복 저장되니, **입력·수정·삭제 시 여러 곳을 동시에 고쳐야 하고 안 그러면 데이터 불일치가 발생**할 위험이 커 — 이게 반정규화의 가장 큰 리스크로 항상 같이 언급.

### **2.1 반정규화의 정의**

반정규화(Denormalization): 정규화된 엔터티, 속성, 관계에 대해 성능 향상과 개발 운영의 단순화를 위해 중복, 통합, 분리 등을 수행하는 데이터 모델링 기법

데이터 무결성이 깨질 수 있는 위험성이 있으므로 신중한 판단 필요

### **2.2 반정규화 적용 시 고려사항**

반정규화 전에 먼저 고려할 사항:

- **뷰(View) 활용**으로 해결 가능한지 검토
- **인덱스 조정**으로 해결 가능한지 검토
- **클러스터링 적용**으로 해결 가능한지 검토
- **파티셔닝이나 캐시 적용** 등 다른 방안 검토
### 1. 뷰 (View) — "임시 가상 테이블"

- **시험 출제 포인트:** 뷰의 특징 중 '보안성'과 '독립성'을 묻는 문제가 많습니다.
    
- **핵심 개념:** 실제 데이터를 저장하지 않는 가상 테이블입니다. 사용자에게 복잡한 조인문을 숨겨 편의성을 주고, 민감한 컬럼(예: 주민번호)을 숨겨서 보여줄 수 있어 **보안**에 좋습니다. 단, `ALTER`로 구조 변경이 안 되고 인덱스를 가질 수 없다는 특징이 단답형으로 자주 나옵니다.
    

### 2. 인덱스 (Index) — "조회 속도 향상 vs 삽입/수정 속도 저하"

- **시험 출제 포인트:** 인덱스를 무조건 많이 만들면 좋은지, 그리고 인덱스의 역효과를 묻습니다.
    
- **핵심 개념:** `SELECT` 성능은 획기적으로 올라가지만, 데이터를 새로 넣거나 바꿀 때(`INSERT`, `UPDATE`, `DELETE`)마다 인덱스 색인 표도 계속 새로 고쳐야 하므로 **오히려 성능이 떨어질 수 있다는 단점**이 단골 문제입니다.
    

### 3. 클러스터링 (Clustering) — "물리적 정렬"

- **시험 출제 포인트:** 클러스터드 인덱스(Clustered Index)와 넌클러스터드 인덱스의 차이점 비교로 나옵니다.
    
- **핵심 개념:** 테이블당 단 **한 개**만 생성할 수 있습니다. 책의 '페이지 번호'처럼 데이터 자체를 알파벳 순이나 숫자 순으로 진짜 하드디스크 상에 정렬해 버리는 방식입니다. 범위(`BETWEEN`)로 조회할 때 압도적으로 빠릅니다.
    

### 4. 파티셔닝 (Partitioning) — "대용량 데이터 분할"

- **시험 출제 포인트:** 파티션의 종류(분할 기준)를 매칭하는 문제가 자주 나옵니다.
    
- **핵심 개념:** 테이블을 논리적으로는 하나지만 물리적으로 여러 개로 쪼갭니다.
    
    - **Range Partitioning (범위):** 날짜나 숫자 범위 기준 (예: 2026년 데이터 따로)
        
    - **List Partitioning (목록):** 특정 값 기준 (예: 서울, 대전, 대구 지역별 따로)
        
    - **Hash Partitioning (해시):** 해시 함수를 이용해 알아서 균등하게 분할

위 방안으로도 해결되지 않을 때 반정규화를 적용

### **2.3 반정규화 기법**

|대상|기법|
|---|---|
|**테이블**|병합 / 분할(수직·수평) / 추가(중복·통계·이력·부분)|
|**컬럼**|중복 컬럼 / 파생 컬럼 / PK 컬럼 추가 등|
|**관계**|중복 관계 추가|

**테이블 반정규화**

- **테이블 병합**: 1:1 관계 또는 1:M 관계의 테이블을 하나로 합침
- **테이블 분할**:
- 수직 분할: 컬럼 단위로 테이블을 나눔 (자주 사용하는 컬럼과 그렇지 않은 컬럼을 분리)
- 수평 분할: 행 단위로 테이블을 나눔 (지역별, 기간별 등으로 분리)
- 테이블 추가:
- 중복 테이블 추가: 다른 업무이거나 서버가 다른 경우 중복 테이블 생성
- 통계 테이블 추가: 통계 데이터를 미리 계산하여 별도 테이블에 저장
- 이력 테이블 추가: 변경 이력을 관리하기 위한 테이블 추가
- 부분 테이블 추가: 자주 사용하는 컬럼만 모아 별도 테이블 생성

**컬럼 반정규화**

- **중복 컬럼** **추가**: 조인 감소를 위해 다른 테이블의 컬럼을 중복 저장
- **파생 컬럼 추가**: 계산값을 미리 저장하여 조회 성능 향상 (예: 합계, 평균)
- 이력 테이블 컬럼 추가: 최신 데이터를 빠르게 조회하기 위한 컬럼 추가
- **PK에 의한 컬럼 추가**: PK에 포함된 정보를 별도 컬럼으로 저장하여 인덱스 접근 편의 제공
- 응용시스템 오동작을 위한 컬럼 추가: 이전 데이터와의 비교를 위한 컬럼

**관계 반정규화**

- 중복 관계 추가: 조인 경로를 단축시키기 위해 이미 존재하는 관계를 중복으로 추가

## **3. 트랜잭션**

### **3.1 트랜잭션의 정의**

**트랜잭션**은 데이터베이스의 상태를 변화시키는 **논리적인 작업 단위**. 즉, 하나 이상의 SQL 문(INSERT, UPDATE, DELETE 등)을 하나의 묶음으로 처리해서, "전부 성공" 또는 "전부 실패" 둘 중 하나로만 처리되도록 보장하는 개념

**대표 예시**: 계좌이체

- A계좌에서 10만원 출금 (UPDATE)
- B계좌에 10만원 입금 (UPDATE)

이 두 작업은 반드시 **둘 다 성공하거나 둘 다 실패**. 출금만 되고 입금이 안 되면 돈이 사라지는 문제 발생. 이걸 하나의 트랜잭션으로 묶어서 관리

### **3.2 트랜잭션의 특성 (ACID)**

- **원자성(Atomicity):** 트랜잭션의 연산은 전부 실행되거나 전부 취소되어야 함 (All or Nothing). 일부만 수행되어서는 안 됨
- **일관성(Consistency):** 트랜잭션 수행 전후로 데이터베이스의 상태가 일관되어야 함. 데이터 무결성 제약 조건을 항상 만족
- **고립성(Isolation):** 하나의 트랜잭션이 실행 중일 때 다른 트랜잭션이 접근할 수 없음. 각 트랜잭션은 독립적으로 수행
- **영속성(Durability):** 트랜잭션이 성공적으로 완료되면 그 결과는 영구적으로 반영. 시스템 장애가 발생하더라도 결과는 보존

### 트랜잭션 제어어 (TCL: Transaction Control Language)

- **COMMIT**: 트랜잭션 작업을 확정(저장)함
- **ROLLBACK**: 트랜잭션 작업을 취소하고 이전 상태로 되돌림
- **SAVEPOINT**: 트랜잭션 내에 저장 지점을 만들어, 그 지점까지만 부분적으로 ROLLBACK 가능하게 함

sql

```sql
INSERTINTO 계좌VALUES('A',-100000);
SAVEPOINT sp1;
INSERTINTO 계좌VALUES('B',100000);

ROLLBACKTO sp1;-- sp1 지점까지만 롤백 (B 입금만 취소)
-- 또는COMMIT;-- 전체 확정
```

### 트랜잭션 격리 수준 (Isolation Level) — 자주 출제!

동시성 문제를 어느 정도 허용할지 정하는 단계. 낮을수록 성능은 좋지만 데이터 정합성 문제가 생길 수 있음

1. **READ UNCOMMITTED**: 커밋 안 된 데이터도 읽음 (**Dirty Read** 발생 가능)
2. **READ COMMITTED**: 커밋된 데이터만 읽음 (대부분 DB의 기본값)
3. **REPEATABLE READ**: 트랜잭션 동안 같은 데이터를 반복 조회해도 항상 같은 결과 보장
4. **SERIALIZABLE**: 완전히 순차적으로 실행되는 것처럼 격리 (가장 엄격, 성능 저하 큼)

#### 발생 가능한 이상 현상

- **Dirty Read**: 커밋되지 않은 데이터를 읽는 문제
- **Non-Repeatable Read(반복 불가능한 읽기)**: 같은 행(row)을 두 번 읽었는데 값이 다른 문제
- **Phantom Read**: 같은 조건으로 조회했는데 없던 행이 생기는 문제

### 이상 현상 발생 정리표

|격리 수준|Dirty Read|Non-Repeatable Read|Phantom Read|
|---|---|---|---|
|READ UNCOMMITTED|발생|발생|발생|
|READ COMMITTED|방지|**발생**|발생|
|REPEATABLE READ|방지|방지|**발생**|
|SERIALIZABLE|방지|방지|방지|

### 핵심 포인트

"REPEATABLE READ는 **있던 행의 값 변경**은 막지만, **새로 생기는 행**까지는 못 막는다"

즉, Non-Repeatable Read는 **UPDATE로 인한 값 변화**, Phantom Read는 **INSERT/DELETE로 인한 행 개수 변화** 문제

### 시험 팁

SQLD에서는 특히

- **ACID의 각 개념을 구분해서 묻는 문제** (예: "다음 설명은 ACID 중 무엇인가?")
- **COMMIT/ROLLBACK/SAVEPOINT의 동작 결과를 SQL 코드로 묻는 문제**
- **격리 수준별 이상 현상 매칭 문제**

### **3.3 트랜잭션과 데이터 모델링**

- 하나의 트랜잭션에 관련된 데이터는 가능하면 하나의 엔터티에 배치
- 트랜잭션의 단위가 데이터 모델의 엔터티 도출에 영향을 줌

## **4. NULL**

### **4.1 NULL의 정의**

NULL: 아직 정의되지 않은 값, 값이 존재하지 않는 상태

NULL ≠ 공백(빈 문자열): NULL은 '값이 없음'이고, 공백은 '길이가 0인 문자열'

NULL ≠ 0: NULL은 '값이 없음'이고, 0은 숫자 0이라는 값

### **4.2 NULL의 특성**

- 산술 연산: NULL이 포함된 산술 연산의 결과는 항상 NULL
    
    예: 100 + NULL = NULL, NULL * 3 = NULL
    
- 비교 연산: NULL과의 비교 결과는 항상 Unknown(알 수 없음)
    
    예: NULL = NULL → Unknown, NULL > 0 → Unknown
    
- 논리 연산: NULL은 3가지 값 논리(Three-Valued Logic)에 따름
    
- TRUE AND Unknown = Unknown
    
- **FALSE AND Unknown = FALSE**
    
- **TRUE OR Unknown = TRUE**
    
- FALSE OR Unknown = Unknown
    
- NOT Unknown = Unknown
    

### **4.3 NULL 관련 함수**

- NVL(Oracle) / ISNULL(SQL Server): NULL을 지정한 값으로 치환
    
    NVL(컬럼, 대체값): 컬럼이 NULL이면 대체값 반환
    
- NULLIF: 두 값이 같으면 NULL 반환, 다르면 첫 번째 값 반환
    
    NULLIF(값1, 값2): 값1 = 값2이면 NULL, 아니면 값1
    
- **COALESCE**는 인자로 주어진 값들을 순서대로 검사해서, **NULL이 아닌 첫 번째 값**을 반환하는 함수.
    
    sql
    
    ```sql
    COALESCE(값1, 값2, 값3,..., 값N)
    ```
    
    - 값1이 NULL이 아니면 → 값1 반환
    - 값1이 NULL이면 → 값2 확인
    - 값2도 NULL이면 → 값3 확인
    - ... 이런 식으로 계속 진행
    - **모든 값이 NULL이면** → NULL 반환
    
    #### 1. 컬럼 값이 NULL일 때 대체값 지정
    
    sql
    
    ```sql
    SELECT 이름,COALESCE(전화번호,'연락처 없음')AS 전화번호FROM 회원;
    ```
    
    → 전화번호가 NULL인 회원은 "연락처 없음"으로 표시
    
    #### 2. 여러 컬럼 중 우선순위대로 값 선택
    
    sql
    
    ```sql
    SELECTCOALESCE(휴대폰번호, 집전화번호, 회사전화번호,'연락처 없음')AS 대표연락처FROM 회원;
    ```
    
    → 휴대폰 있으면 휴대폰, 없으면 집전화, 그것도 없으면 회사전화 순으로 선택
    
    #### 3. 집계 함수와 함께 사용 (NULL로 인한 계산 오류 방지)
    
    sql
    
    ```sql
    SELECT 상품명,COALESCE(할인율,0)AS 할인율FROM 상품;
    ```
    
    → 할인율이 NULL이면 0으로 처리해서 계산 오류 방지
    
    ### NVL과의 비교 (시험에 자주 나옴!)
    
    |구분|NVL|COALESCE|
    |---|---|---|
    |표준 여부|Oracle 전용 함수 (비표준)|**ANSI SQL 표준** 함수|
    |인자 개수|2개만 가능: `NVL(값, 대체값)`|**여러 개 가능**: `COALESCE(값1, 값2, ..., 값N)`|
    |지원 DBMS|Oracle|Oracle, SQL Server, MySQL 등 대부분 지원|
    
    sql
    
    ```sql
    -- NVL: 인자 2개만NVL(전화번호,'없음')
    
    -- COALESCE: 여러 개 가능, 더 유연함COALESCE(휴대폰, 집전화, 회사전화,'없음')
    ```
    
    ### CASE문과의 관계
    
    사실 COALESCE는 내부적으로 CASE문과 동일하게 동작해요.
    
    sql
    
    ```sql
    -- 이 COALESCE는COALESCE(A, B, C)
    
    -- 이 CASE문과 완전히 동일함
    
    CASE
    WHEN A IS NOT NULL THEN A
    WHEN B IS NOT NULL THEN B
    ELSE C
    END
    ```
    
    ### 시험 포인트 정리
    
    1. **NULL이 아닌 첫 번째 값**을 반환한다는 것 (순서 중요!)
    2. 모든 인자가 NULL이면 **결과도 NULL**
    3. **NVL과 차이점**: COALESCE는 표준 함수이고 인자를 여러 개 받을 수 있음
    4. CASE WHEN 구문으로 치환 가능하다는 것

### **4.4 집계 함수와 NULL**

**집계 함수는 기본적으로 NULL을 계산에서 제외한다** (단, COUNT(*)만 예외)

- SUM, AVG, COUNT(컬럼), MAX, MIN: NULL 값을 제외하고 연산
- COUNT(*): NULL 포함 전체 행 수를 카운트
- AVG 주의: NULL을 제외한 행의 평균이므로, 전체 행의 평균과 다를 수 있음

#### 1. COUNT(*) vs COUNT(컬럼) — 결과가 다를 수 있음

sql

```sql
-- 예시 데이터: 학생 5명, 그 중 2명은 점수가 NULL(미응시)

COUNT(*)-- 5  (NULL 포함 전체 행)

COUNT(점수)-- 3  (NULL 제외한 값이 있는 행만)
```

→ **똑같은 테이블에서 COUNT(*)와 COUNT(컬럼)의 결과가 다를 수 있다**는 게 시험 포인트.

#### 2. AVG의 함정 — "전체 평균"이 아니다

예를 들어 학생 5명의 점수가 다음과 같다.

```
90, 80, NULL, 70, NULL
```

**사람들이 착각하는 계산** (전체 5명 기준):

```
(90 + 80 + 70) / 5 = 48
```

**실제 AVG의 계산** (NULL 제외한 3명 기준):

```
AVG(점수) = (90 + 80 + 70) / 3 = 80
```

→ **분모가 전체 행 수가 아니라, NULL이 아닌 값의 개수**라는 게 핵심이에요.

### 정리: **"학생 5명 중 2명이 시험을 안 봐서 점수가 NULL이다. AVG(점수)의 결과는?"**

- ❌ 틀린 생각: 전체 5명으로 나눔
- ✅ 맞는 생각: NULL이 아닌 3명으로만 나눔

### 한 줄 요약

> **SUM, AVG, MAX, MIN, COUNT(컬럼)은 NULL을 무시하고 계산하지만, COUNT(*)만 NULL 포함 전체 행을 센다. 그래서 AVG는 "전체 행 대비 평균"이 아니라 "NULL 제외 행 대비 평균"이다.**

만약 **"NULL을 0으로 처리한 평균"을 구하고 싶다면** COALESCE나 NVL을 같이 써서 명시적으로 처리

sql

```sql
AVG(COALESCE(점수,0))-- NULL을 0으로 바꾼 후 전체 5명 기준 평균
```

## **5. 본질식별자 vs. 인조식별자**

### 가장 중요한 한 줄

> **본질식별자 = 업무에서 원래 있던 값 / 인조식별자 = DB가 관리 편하려고 인위적으로 만든 값**

|구분|본질식별자|인조식별자|
|---|---|---|
|생성 방식|업무 규칙에서 자연 발생|시스템이 자동 생성 (시퀀스, Auto Increment)|
|의미|그 값 자체로 의미 있음|의미 없음, 그냥 유일성 보장용|
|예시|주민등록번호, 사업자등록번호|주문번호, 회원번호(시퀀스)|

#### "인조식별자를 PK로 쓰면 끝나는 게 아니다"

_PK = 테이블에서 각 행(row)을 유일하게 구분해주는 값, 예) 학번) pk 특징 : 고유함. NOT NULL 학번없는 학생은 존재할 수 없음_

이게 이 챕터의 핵심 함정이에요. 예시: **수강신청** 케이스

```
수강신청 테이블PK: 수강신청번호 (인조식별자, 시퀀스로 자동 생성)
```

이렇게만 하면 문제가 생겨요:

sql

```sql
-- 이런 중복이 그냥 다 들어가버림!
수강신청번호1, 학번2020001, 과목 DB개론
수강신청번호2, 학번2020001, 과목 DB개론  ← 같은 학생이 같은 과목 또 신청!
```

**왜냐하면** 인조식별자(수강신청번호)는 그냥 자동으로 매겨지는 순번일 뿐이라, **"학번+과목번호가 겹치면 안 된다"는 업무 규칙을 전혀 막아주지 못하기** 때문.

#### 해결책

sql

```sql
-- 본질식별자(학번+과목번호)에 UNIQUE 제약조건을 별도로 걸어줘야 함

UNIQUE(학번, 과목번호)
```

**즉, 인조식별자를 PK로 쓰더라도, 본질식별자는 UNIQUE 제약으로 반드시 별도 관리해야 한다**는 게 핵심

### 인조식별자를 언제 쓰면 좋은가 (3가지 상황)

1. **본질식별자가 컬럼이 너무 많은 복합키**일 때 → 하나의 인조 컬럼으로 단순화
2. *자식 엔터티로 식별자가 계속 전이(상속)**되어 관리가 복잡할 때 → 예: 부모 PK가 3개 컬럼이면 자식, 손자까지 계속 3개씩 물려받아 지저분해짐
3. **본질식별자 값이 너무 길 때** → 인덱스 크기/성능에 영향 주니까 짧은 인조키로 대체

본질식별자(학번+과목번호) 이렇게 + 표시

이건 **두 개 이상의 컬럼을 합쳐서 하나의 식별자로 쓴다**는 뜻.  **복합키(Composite Key)**

### **왜 하나만으로는 안 될까?**

수강신청 상황을 생각해보면:

|**학번**|**과목번호**|
|---|---|
|2020001|DB개론|
|2020001|자료구조|
|2020002|DB개론|

**학번만으로는 유일하지 않다.**

- 2020001번 학생은 DB개론도 신청했고, 자료구조도 신청. 즉 학번 2020001이 테이블에 **두 번** 등장.

**과목번호만으로도 유일하지 않음.**

- DB개론은 2020001번 학생도 듣고, 2020002번 학생도 들어요. 즉 "DB개론"도 **두 번** 나옴.

### **그럼 언제 유일해지나?**

**학번 + 과목번호를 묶으면** 그제서야 유일.

```
(2020001, DB개론)   ← 이 조합은 딱 하나
(2020001, 자료구조) ← 이 조합도 딱 하나
(2020002, DB개론)   ← 이것도 딱 하나
```

### **SQL로 표현하면**

sql

```sql
CREATETABLE 수강신청(   
 학번INT,    
 과목번호INT,    
 신청일자DATE,
 PRIMARYKEY(학번, 과목번호)-- 두 컬럼을 묶어서 PK로 지정!
);
```

### **5.1 본질식별자**

본질식별자(Natural Identifier): 업무에 의해 자연적으로 만들어진 식별자

업무적 의미를 가지고 있어 데이터의 의미를 직관적으로 파악 가능

예: 주민등록번호, 사업자등록번호, 수강신청(학번+과목번호)

### **5.2 인조식별자**

인조식별자(Surrogate Identifier): 인위적으로 부여한 식별자

업무적 의미는 없으나 시스템 관리를 위해 생성

주로 자동 증가(Auto Increment) 번호나 시퀀스를 사용

예: 수강신청번호(시퀀스), 주문번호(자동채번)

### **5.3 인조식별자 사용 시 고려사항**

- 불필요한 인조식별자 남용 주의: 본질식별자만으로 충분한 경우 인조식별자 추가 지양
    
- 중복 데이터 발생 위험: 인조식별자만으로는 업무적 중복을 방지하기 어려움
    
    예: 수강신청번호만 PK로 설정 시, 동일 학생이 같은 과목을 중복 신청 가능. 본질식별자를 유니크 제약조건으로 설정하거나 PK에 포함 필요
    
- 불필요한 인덱스 증가: 인조식별자를 추가하면 관련 인덱스도 추가 필요
    
- 인조식별자가 적합한 경우:
    
- 본질식별자가 너무 많은 컬럼으로 구성된 복합식별자일 때
    
- 자식 엔터티로 식별자가 계속 전이되어 관리가 복잡한 경우
    
- 본질식별자의 값이 너무 길어 인덱스 성능에 영향을 줄 때


## **6. 계층형 데이터 모델**

### 6.1 계층형 데이터 모델이란?

> **한 테이블 안에서 "자기 자신을 참조하는 FK"를 가지는 구조**

조직도 예시

|사원번호(PK)|이름|관리자번호(FK)|
|---|---|---|
|1|김대표|NULL|
|2|이부장|1|
|3|박과장|2|
|4|최사원|2|

**핵심 포인트**: "관리자번호" 컬럼은 **같은 테이블(사원 테이블)의 사원번호를 가리키는 FK**.

```
김대표(1)  └─ 이부장(2)  ← 관리자번호가 1 (김대표를 가리킴)        ├─ 박과장(3)  ← 관리자번호가 2 (이부장을 가리킴)        └─ 최사원(4)  ← 관리자번호가 2 (이부장을 가리킴)
```

→ **다른 테이블 참조하는 일반 FK와 달리, 자기 자신의 테이블을 참조**. **Self-Relationship(재귀관계)**

### 6.2 셀프 조인 (Self Join)

- *"박과장의 관리자가 누구야?"**를 조회

같은 테이블을 **두 번 불러와서**, 하나는 "사원 입장", 하나는 "관리자 입장"으로 취급.

sql

```sql
SELECT    
	E.이름AS 사원이름,    
	M.이름AS 관리자이름
FROM 사원 E-- E = Employee (사원 역할)
JOIN 사원 MON E.관리자번호= M.사원번호-- M = Manager (관리자 역할)
```

**결과:**

|사원이름|관리자이름|
|---|---|
|이부장|김대표|
|박과장|이부장|
|최사원|이부장|

→ 같은 테이블인데 `E`, `M`이라는 **별칭(Alias)**을 붙여서 마치 서로 다른 두 테이블인 것처럼 조인하는 거예요. 그래서 "셀프(self) 조인"이라고 부름.

### 6.3 계층형 SQL

**전체 조직도를 위에서 아래로 쭉 펼쳐서 보고 싶다면** 셀프 조인만으로는 부족(한 단계씩만 보이니까). 이럴 땐 **재귀적으로 계속 타고 내려가는** 전용 문법이 필요.

#### Oracle 방식

sql

```sql
SELECTLEVEL, 사원번호, 이름, 관리자번호
FROM 사원
START WITH 관리자번호 IS NULL-- 루트(최상위, 여기선 김대표)부터 시작
CONNECT BY PRIOR 사원번호= 관리자번호-- 부모→자식 방향으로 계속 타고 내려감
ORDER SIBLINGS BY 이름;-- 같은 레벨끼리는 이름순 정렬
```

|개념|의미|
|---|---|
|START WITH|"어디서부터 출발할래?" (보통 최상위 = 부모가 NULL인 행)|
|CONNECT BY PRIOR|"부모→자식 방향으로 계속 연결해서 내려가라"|
|LEVEL|김대표=1, 이부장=2, 박과장/최사원=3 처럼 **깊이**를 자동으로 알려줌|
|ORDER SIBLINGS BY|형제(같은 레벨) 노드끼리만 정렬 (전체 순서가 깨지지 않게)|

#### SQL Server 방식 (CTE 재귀쿼리)

sql

```sql
WITH RECURSIVE 조직도 AS(-- 1) 앵커(anchor): 시작점 (루트)
  SELECT 사원번호, 이름, 관리자번호,1AS 레벨
  FROM 사원
  WHERE 관리자번호 IS NULL
   
  UNION ALL

  -- 2) 재귀(recursive): 자기 자신을 다시 참조하며 한 단계씩 내려감
  SELECT E.사원번호, E.이름, E.관리자번호, O.레벨+1
  FROM 사원 E
  JOIN 조직도 O ON E.관리자번호= O.사원번호

)

SELECT*FROM 조직도;
```

→ **CTE가 자기 자신(조직도)을 UNION ALL로 계속 참조**하면서, 더 이상 자식이 없을 때까지 반복해서 내려가는 구조.

### 6.4 설계 시 고려사항

#### 1) 순환 참조(Cycle) 금지

```
A의 관리자가 B, B의 관리자가 A ← 이런 구조는 안 됨!
```

→ 이러면 CONNECT BY나 재귀쿼리가 **무한 루프**에 빠질 수 있음.

#### 2) 루트 노드의 부모키는 NULL

가장 위(김대표)는 관리자가 없으니까, 관리자번호가 **NULL**이어야 해요. 이게 바로 `START WITH 관리자번호 IS NULL`의 근거.

#### 3) 깊이가 깊어지면 성능 저하

계층이 10단계, 20단계로 깊어지면 매번 재귀적으로 타고 내려가야 해서 느려질 수 있어요.

#### 4) 해결책: PATH 컬럼 추가

```
사원번호 | 이름   | PATH1        | 김대표 | /12        | 이부장 | /1/23        | 박과장 | /1/2/34        | 최사원 | /1/2/4
```

→ 미리 "이 사람까지 오는 전체 경로"를 문자열로 저장해두면, `LIKE '/1/2/%'` 같은 조건으로 **재귀 계산 없이 한 방에** "이부장 아래 전체 조직원"을 찾을 수 있어요. **일종의 반정규화(파생 컬럼 추가) 적용 사례**.

### 정리 한 줄

> **셀프 조인 = 한 단계(부모-자식)만 조회**, **START WITH/CONNECT BY, 재귀 CTE = 전체 계층을 한 번에 조회**

### **6.1 계층형 데이터 모델의 정의**

엔터티 내에서 자기 자신과의 관계(Self-Relationship)를 가지는 데이터 모델

조직도, 카테고리, 메뉴 구조 등 트리(Tree) 구조의 데이터를 표현

하나의 테이블 내에서 상위-하위 관계를 부모키(FK)로 표현

### **6.2 셀프 조인 (Self Join)**

셀프 조인: 같은 테이블을 두 번 이상 참조하여 조인하는 방식

계층형 데이터에서 상위-하위 관계를 조회할 때 사용

테이블에 별칭(Alias)을 부여하여 마치 다른 테이블처럼 사용

예: 사원 테이블에서 사원과 해당 사원의 관리자 정보를 함께 조회

### **6.3 계층형 SQL**

Oracle의 계층형 쿼리:

- START WITH: 계층의 시작점(루트) 지정
- CONNECT BY PRIOR: 부모-자식 관계의 방향 지정
- LEVEL: 현재 계층의 깊이(레벨) 반환
- ORDER SIBLINGS BY: 같은 레벨 내에서 정렬

SQL Server의 계층형 쿼리:

- CTE(Common Table Expression)와 재귀 쿼리를 사용하여 구현
- WITH RECURSIVE 구문 활용

### **6.4 계층형 데이터 모델 설계 시 고려사항**

- 순환 참조(Cycle)가 발생하지 않도록 주의
- 루트 노드의 부모키는 NULL로 설정
- 계층의 깊이가 깊어지면 조회 성능 저하 가능
- 필요 시 경로(PATH) 컬럼을 추가하여 탐색 성능 향상

### **정리 및 핵심 포인트**

1. **정규화**는 데이터 중복을 제거하고 이상 현상을 방지하기 위해 테이블을 분해하는 과정이며**, 1NF(원자값) → 2NF(부분종속 제거) → 3NF(이행종속 제거) → BCNF(결정자=후보키)** 순으로 진행
2. **반정규화**는 성능 향상을 위해 정규화된 모델에 중복/통합/분리를 적용하는 기법이며, 반드시 다른 대안(뷰, 인덱스, 파티셔닝 등)을 먼저 검토한 후 적용해야 합니다.
3. **트랜잭션**은 ACID(원자성, 일관성, 고립성, 영속성) 특성을 가지며, 전부 수행되거나 전부 취소되는 논리적 작업 단위입니다.
4. NULL은 '값이 없음'을 의미하며, 산술 연산 결과는 NULL, 비교 연산 결과는 Unknown이 됩니다. NVL, NULLIF, COALESCE 등의 함수로 처리합니다.
5. 집계 함수(SUM, AVG, MAX, MIN, COUNT(컬럼))는 NULL을 제외하고 연산하며, **COUNT(*)만 NULL을 포함하여 전체 행 수를 계산**합니다.
6. **본질식별자**는 업무적으로 자연 발생한 식별자이고, **인조식별자**는 시스템 편의를 위해 인위적으로 부여한 식별자입니다. 인조식별자 남용 시 중복 데이터 발생 위험이 있습니다.
7. 계층형 데이터 모델은 자기 자신과의 관계를 가지며, 셀프 조인이나 계층형 쿼리(START WITH/CONNECT BY, CTE)를 통해 조회합니다.
8. 정규화는 입력/수정/삭제 성능을 향상시키고, 반정규화는 조회 성능을 향상시키는 대신 데이터 무결성 위험이 있으므로 균형 있는 적용이 중요합니다.

### **1.1 SELECT 문과 SELECT DISTINCT**

### 실제 데이터로 보면 이런 느낌

|EMPNO|ENAME|JOB|SAL|

|7369|SMITH|CLERK|800|

### 1.3 산술 연산자 — NVL과 함께 나오는 이유

여기서 진짜 중요한 포인트는 앞서 배운 **NULL 개념**과 바로 연결.

실제 EMP 예제 테이블

|EMPNO|ENAME|JOB|**SAL**|**COMM**|
|---|---|---|---|---|
|7499|ALLEN|SALESMAN|1600|300|
|7521|WARD|SALESMAN|1250|500|
|7566|JONES|MANAGER|2975|NULL|

- **SAL** = Salary(급여)의 줄임말 → 월급
- **COMM** = Commission(커미션/수당)의 줄임말 → 영업직 등이 실적에 따라 추가로 받는 인센티브. **관리자나 사무직은 커미션이 없어서 NULL인 경우가 많음.**

sql

```sql
(SAL+ NVL(COMM,0))AS TOTAL_COMP
```

**왜 NVL을 꼭 써야 할까요?** 만약 COMM이 NULL인 사원이 있는데 그냥 `SAL + COMM`이라고 하면:

```
SAL(300) + COMM(NULL) = NULL   ← 결과가 NULL이 되어버림!
```

→ **NULL이 포함된 산술 연산의 결과는 무조건 NULL**. (COALESCE 배우실 때 나왔던 원리와 완전히 동일) COMM이 없는 사원은 총액이 그냥 NULL로 나와버리는 오류를 막기 위해 `NVL(COMM, 0)`으로 먼저 0 처리

### 대응관계로 정리하면

|프로그래밍|SQL (NVL)|
|---|---|
|`dict.get('key', default)`|`NVL(컬럼, 기본값)`|

### 문자 합성 연산자 — DBMS별 차이 (시험 단골)

|DBMS|문자열 합성 연산자|
|---|---|
|Oracle|`|
|SQL Server|`+`|
|공통(표준)|`CONCAT()` 함수|

sql

```sql
-- Oracle'ABC'||'DEF'-- 결과: ABCDEF

-- SQL Server'ABC'+'DEF'-- 결과: ABCDEF
```

**주의할 점**: SQL Server에서 `+`는 **숫자 덧셈에도 쓰고, 문자열 합성에도 쓰기 때문에** 헷갈리기 쉽다

Oracle은 `||`(문자합성)와 `+`(산술연산)가 명확히 구분돼서 오히려 헷갈릴 일이 적음.

### 전체 흐름 한눈에

```
SELECT [ALL/DISTINCT] 컬럼(연산/별칭 포함) FROM 테이블;
```

sql

```sql
SELECT DISTINCT ENAME ||' is a '|| JOB AS "사원 정보" 
FROM EMP;
```

→ 중복제거(DISTINCT) + 문자합성(||) + 별칭(AS, 큰따옴표) 이 다 한 문장에 조합.

### 정리

> **DISTINCT는 "합성 연산이 다 끝난 후의 최종 결과값" 기준으로 중복을 판단한다**

### **1.3 산술 연산자와 문자 합성 연산자**

산술 연산자

- +: 덧셈
- : 뺄셈
- : 곱셈
- /: 나눗셈
- %: 나머지 (오라클에서는 MOD 함수 사용)

- - 월급과 커미션 합산

**SELECT** ENAME, SAL, COMM, (SAL + NVL(COMM, 0)) AS TOTAL_COMP **FROM** EMP;

- - 월급에서 커미션 차감

SELECT ENAME, SAL, COMM, (SAL - NVL(COMM, 0)) AS NET_SAL FROM EMP;

- - 연봉 계산 (월급 * 12)

SELECT ENAME, SAL, (SAL * 12) AS ANNUAL_SAL FROM EMP;

- - 주급 계산 (월급 / 4)

SELECT ENAME, SAL, (SAL / 4) AS WEEKLY_SAL FROM EMP;

- - 짝수/홀수 판별 (오라클)

SELECT ENAME, EMPNO,

CASE WHEN MOD(EMPNO, 2) = 0 THEN '짝수' ELSE '홀수' END AS ID_TYPE

FROM EMP;

문자 합성 연산자

- 오라클: || (두 개의 버티컬 라인)
    
- SQL Server: + 연산자
    
- - **오라클**에서 문자열 합성

SELECT ENAME **||** ' is a ' **||** JOB AS EMPLOYEE_INFO FROM EMP;

- - **SQL Server**에서 문자열 합성

SELECT ENAME **+** ' is a ' **+** JOB AS EMPLOYEE_INFO FROM EMP;

- - **CONCAT 함수** 사용 (공통)

SELECT CONCAT(ENAME, CONCAT(' is a ', JOB)) AS EMPLOYEE_INFO FROM EMP;

## 이 표에서 핵심 포인트

전부 다 알아야 하지만, 시험에서 특히 **함정으로 잘 나오는 것 3가지**

### 1. ROUND vs TRUNC — 가장 중요한 비교 (단골 출제)

sql

```sql
ROUND(3.148,2)  →3.15-- 세번째 자리 8이 크니까 반올림해서 올라감
TRUNC(3.148,2)  →3.14-- 8이든 뭐든 상관없이 그냥 뒷자리 버림
```

**핵심 암기**:

> **ROUND = "반올림 판단"이 들어감 (5 이상이면 올림)**
> 
> **TRUNC = "무조건 자르기", 판단 없음**

### 2. CEIL vs FLOOR — 음수에서 헷갈림 주의 (단골 함정)

sql

```sql
CEIL(4.3)   →5-- 4.3보다 크거나 같은 최소 정수
FLOOR(4.7)  →4-- 4.7보다 작거나 같은 최대 정수
```

**근데 음수가 나오면 함정**이 생겨요.

sql

```sql
CEIL(-4.3)   →-4-- 주의! -5 아님. -4.3보다 크거나 같은 정수 중 최소값은 -4
FLOOR(-4.3)  →-5-- 주의! -4 아님. -4.3보다 작거나 같은 정수 중 최대값은 -5
```

### 3. MOD — 나머지만 구함

sql

```sql
MOD(17,5) → 2  -- 17 ÷ 5 = 3 나머지 2
```

앞서 배웠던 것 기억나실 텐데, **Oracle은 `%` 대신 MOD 함수를 씁니다.** (SQL Server 등은 `%` 사용 가능)

#### 1. 손익 상태를 한눈에 분류할 때(SIGN)

sql

```sql
SELECT 상품명, 이익금액,
	CASE SIGN(이익금액)
		WHEN 1 THEN '이익'
		WHEN -1 THEN '손실'
		WHEN 0 THEN'본전' 
	END AS 손익상태
FROM 매출;
```

→ 이익금액이 정확히 얼마인지는 중요하지 않고, **"이익이냐 손실이냐 본전이냐"** 만 빠르게 분류

#### 2. 증감 추세 판단 (전월 대비 증가/감소)

sql

```sql
SELECT 월, 매출액,(매출액- LAG(매출액)OVER(ORDER BY 월))AS 증감,    
	SIGN(매출액- LAG(매출액)OVER(ORDER BY 월))AS 증감방향
FROM 매출;
```

→ "지난달보다 늘었는지 줄었는지"만 딱 필요할 때, 복잡한 증감액 대신 **+1(증가) / -1(감소) / 0(동일)** 로 심플하게 표현.

### LAG 함수란?

> **LAG(컬럼) OVER (ORDER BY 정렬기준) = "바로 이전 행의 값"을 가져오는 함수**

말이 어려우니 예시 테이블로 바로 볼게요.

### 예시 데이터: 매출 테이블

|월|매출액|
|---|---|
|1월|100|
|2월|150|
|3월|120|
|4월|200|

### LAG(매출액) OVER (ORDER BY 월) 적용해보기

**LAG는 "월 순서대로 정렬했을 때, 한 칸 위(이전 행)의 매출액을 가져와라"** 라는 뜻이에요.

|월|매출액|LAG(매출액) = 이전 달 매출액|
|---|---|---|
|1월|100|**NULL** (1월은 이전 달이 없으니까)|
|2월|150|100 (1월 매출액)|
|3월|120|150 (2월 매출액)|
|4월|200|120 (3월 매출액)|

→ 딱 한 칸씩 **"위에서 아래로 밀려서"** 값이 채워지는 거예요. 엑셀에서 "이전 셀 값 가져오기"랑 똑같은 느낌

### 이제 전체 쿼리 계산해보기

sql

```sql
(매출액- LAG(매출액)OVER(ORDERBY 월))AS 증감
```

→ **"이번 달 매출액 - 지난 달 매출액"** = 얼마나 늘었는지/줄었는지

|월|매출액|이전달 매출|증감 (매출액-이전달)|
|---|---|---|---|
|1월|100|NULL|**NULL** (비교할 이전 달 없음)|
|2월|150|100|150-100 = **+50**|
|3월|120|150|120-150 = **-30**|
|4월|200|120|200-120 = **+80**|

### 여기에 SIGN까지 씌우면

sql

```sql
SIGN(매출액- LAG(매출액)OVER(ORDERBY 월))AS 증감방향
```

→ 앞서 배운 SIGN이 이 증감값의 **부호만** 뽑아줘요.

|월|증감|증감방향 (SIGN)|
|---|---|---|
|1월|NULL|NULL|
|2월|+50|**1** (증가)|
|3월|-30|**-1** (감소)|
|4월|+80|**1** (증가)|

### 최종 결과 (전체 쿼리 실행 결과)

|월|매출액|증감|증감방향|
|---|---|---|---|
|1월|100|NULL|NULL|
|2월|150|50|1|
|3월|120|-30|-1|
|4월|200|80|1|

### 정리 한 줄

> **LAG = "한 칸 위(이전) 값 가져오기"** → 그 값을 현재 값에서 빼면 "증감액" → 거기에 SIGN을 씌우면 "증가/감소 여부"만 딱 나옴

### **3.2 연산자 종류와 사용법**

- **<>, != , ^= : 같지 않다**
- > , <, >=, <=: 크다, 작다, 크거나 같다, 작거나 같다
    
- - 급여가 3000인 직원

**SELECT** ENAME, JOB, SAL **FROM** EMP **WHERE** SAL = 3000;

- - 급여가 3000이 아닌 직원

SELECT ENAME, JOB, SAL FROM EMP WHERE SAL **<>** 3000;

- - 급여가 2000 이상인 직원

SQL 연산자

- BETWEEN a AND b: a와 b 사이의 값
- IN (list): 목록에 있는 값 중 하나와 일치
- LIKE '패턴': 패턴과 일치하는 문자열
- NOT BETWEEN, NOT IN, IS NOT NULL: 위 조건들의 부정형
- IS NULL: NULL 값

|구분|의미|
|---|---|
|`IS NULL`|**조건문**. "NULL인지 아닌지" 판별 (참/거짓 반환)|
|`ISNULL(A, B)`|**함수**(SQL Server). A가 NULL이면 B로 바꿔줌 — 사실 Oracle의 **NVL**과 똑같은 함수예요!|

sql

```sql
-- IS NULL: 조건 판별
WHERE COMM IS NULL-- COMM이 NULL인 행만 조회
-- ISNULL: 값 대체 함수 (SQL Server, NVL과 동일한 역할)
SELECT ISNULL(COMM,0) FROM EMP-- COMM이 NULL이면 0으로 바꿔서 보여줌
```

> **IS NULL(공백) = 조건/판별 / ISNULL(함수, SQL Server) = NVL과 같은 대체 함수**

## **4. GROUP BY 절, HAVING 절, ORDER BY 절**

### **4.1 집계 함수 개요**

집계 함수는 여러 행의 데이터를 하나의 결과값으로 집계하는 데 사용되는 함수입니다.

- SUM: 합계
- AVG: 평균
- COUNT: 개수
- MAX: 최댓값
- MIN: 최솟값
- - 전체 급여 합계

SELECT SUM(SAL) FROM EMP;

- - 평균 급여

SELECT AVG(SAL) FROM EMP;

- - 직원 수

SELECT COUNT(*) FROM EMP;

- - 최고 급여

SELECT MAX(SAL) FROM EMP;

- - 최저 급여

SELECT MIN(SAL) FROM EMP;

### **4.2 GROUP BY 절**

데이터를 그룹화하여 집계 함수를 각 그룹에 적용할 수 있게 합니다.

- - 부서별 급여 합계와 평균

SELECT DEPTNO, SUM(SAL) AS TOTAL_SAL, AVG(SAL) AS AVG_SAL

FROM EMP

GROUP BY DEPTNO;

- - 부서와 직무별 급여 합계

SELECT DEPTNO, JOB, SUM(SAL) AS TOTAL_SAL

FROM EMP

GROUP BY DEPTNO, JOB

ORDER BY DEPTNO;

주의사항:

- GROUP BY 절에서는 ALIAS를 사용할 수 없습니다.
- SELECT 절에 GROUP BY 절에 없는 칼럼은 사용할 수 없습니다(집계 함수 제외).

### **4.3 HAVING 절**

GROUP BY로 그룹화된 결과에 조건을 적용할 때 사용합니다.

- - 평균 급여가 2000을 초과하는 부서 조회

SELECT DEPTNO, AVG(SAL) AS AVG_SAL

FROM EMP

GROUP BY DEPTNO

HAVING AVG(SAL) > 2000;

- - 직원이 3명 이상인 직무 조회

SELECT JOB, COUNT(*) AS EMP_COUNT

FROM EMP

GROUP BY JOB

HAVING COUNT(*) >= 3;

WHERE 절과 HAVING 절의 차이:

- WHERE: 그룹화 전 개별 행을 필터링 (집계 함수 사용 불가)
- HAVING: 그룹화 후 그룹을 필터링 (집계 함수 사용 가능)
- - WHERE와 HAVING 절 함께 사용

SELECT DEPTNO, AVG(SAL) AS AVG_SAL

FROM EMP

WHERE SAL >= 1000 -- 개별 급여 1000 이상인 직원만 대상

GROUP BY DEPTNO

HAVING AVG(SAL) > 2000; -- 그 중 평균 급여 2000 초과 부서만 선택

### **4.4 ORDER BY 절**

쿼리 결과를 특정 칼럼 기준으로 정렬할 때 사용합니다.

- - 오름차순 정렬 (ASC 생략 가능)

SELECT ENAME, SAL

FROM EMP

ORDER BY SAL ASC;

- - 내림차순 정렬

SELECT ENAME, SAL

FROM EMP

ORDER BY SAL DESC;

- - 여러 열로 정렬 (부서번호 오름차순, 동일 부서 내 급여 내림차순)

SELECT ENAME, JOB, SAL, DEPTNO

FROM EMP

ORDER BY DEPTNO ASC, SAL DESC;

SQL 실행 순서:

1. FROM → 2. WHERE → 3. GROUP BY → 4. HAVING → 5. SELECT → 6. ORDER BY

## **5. JOIN**

### **5.1 JOIN의 개념**

JOIN은 두 개 이상의 테이블에서 관련된 데이터를 결합하여 새로운 결과 집합을 생성하는 과정입니다. 조인의 기본 목적은 테이블 간의 관계를 활용하여 필요한 정보를 효율적으로 검색하는 것입니다.

### **5.2 JOIN의 기본 문법**

SELECT 별칭1.칼럼명, 별칭2.칼럼명

FROM 테이블1 AS 별칭1

JOIN 테이블2 AS 별칭2

ON 별칭1.칼럼명 = 별칭2.칼럼명;

### **5.3 JOIN의 종류**

## INNER JOIN 문장 구조 분해

sql

```sql
SELECT E.ENAME, D.DNAME
FROM EMP E
INNER JOIN DEPT D 
ON E.DEPTNO= D.DEPTNO;
```

### 조각별로 뜯어보기

#### 조각 1: `FROM EMP E`

```
FROM  EMP  E     
		   ↑   ↑   
	  테이블 별칭
```

→ "EMP 테이블을 쓸 건데, 앞으로 이걸 **E**라고 줄여 부를게"

#### 조각 2: `INNER JOIN DEPT D`

```
INNER JOIN DEPT D           
						↑   ↑         
				테이블  별칭
```

→ "그리고 DEPT 테이블도 같이 쓸 건데, 이건 **D**라고 줄여 부를게"

**여기까지가 "어떤 테이블 두 개를 쓸 건지" 선언하는 부분**

```
FROM EMP E INNER JOIN DEPT D

= "EMP(별명 E)랑 DEPT(별명 D), 이 두 테이블을 합칠 거야"
```

#### 조각 3: `ON E.DEPTNO = D.DEPTNO`

이게 핵심이에요. **"어떻게 두 테이블을 짝지을지"** 규칙이에요.

```
ON E.DEPTNO = D.DEPTNO   
      ↑           ↑ 
  E테이블의    D테이블의   
 DEPTNO 컬럼  DEPTNO 컬럼
```

→ "EMP의 DEPTNO 값이랑, DEPT의 DEPTNO 값이 **똑같은 행끼리** 짝지어라"

#### 조각 4: `SELECT E.ENAME, D.DNAME`

짝지어진 결과에서 **어떤 컬럼만 보여줄지** 고르는 부분이에요.

```
E.ENAME  → EMP 테이블의 ENAME 컬럼
D.DNAME  → DEPT 테이블의 DNAME 컬
```

→ 왜 `E.`, `D.`를 붙였을까요? **두 테이블이 합쳐졌으니, "이 컬럼이 어느 테이블 거야?"를 명확히 구분**해줘야 하기 때문

### 전체를 한글 순서로 다시 배치해보면

SQL은 실행 순서가 작성 순서랑 달라서 헷갈리는데, **이해하기 쉬운 순서**로 다시 나열해볼게요.

```
1단계: **EMP** 테이블(별명 E)이랑 **DEPT** 테이블(별명 D)을 준비해
2단계: 그중에서 **E.DEPTNO**랑 **D.DEPTNO**가 같은 행끼리만 짝지어
3단계: 짝지어진 결과에서 **E.ENAME**이랑 **D.DNAME**만 뽑아서 보여줘
```

### 최종 결과

|ENAME|DNAME|
|---|---|
|SMITH|RESEARCH|
|ALLEN|SALES|
|FORD|RESEARCH|

### 정리: 문장 구조를 4개 조각으로 외우기

```
SELECT [보여줄 컬럼]
FROM [테이블1] [별칭1]
INNER JOIN [테이블2] [별칭2]
ON [별칭1.짝지을컬럼]= [별칭2.짝지을컬럼];
```

이 뼈대만 기억해두면, 어떤 JOIN 쿼리를 봐도 "아 여기가 테이블 지정하는 부분, 여기가 짝짓는 조건, 여기가 보여줄 컬럼 고르는 부분"이라고 분해해서 읽을 수 있어요.

두 테이블에서 공통된 값을 가진 행만 반환합니다.

SELECT E.EMPNO, E.ENAME, D.DNAME

FROM EMP E INNER JOIN DEPT D

ON E.DEPTNO = D.DEPTNO;

**LEFT OUTER JOIN**

### `*` 는 "ALL"이 아니라 "모든 컬럼"이라는 뜻

|키워드|의미|역할|
|---|---|---|
|`SELECT *`|**모든 컬럼**을 보여줘|컬럼 선택|
|`SELECT ALL`|중복 있어도 다 보여줘 (DISTINCT의 반댓말)|중복 처리 방식|

→ `*`는 "가로(컬럼)를 얼마나 보여줄지"에 관한 거고, `ALL`은 "중복된 행을 지울지 말지"에 관한 거. **전혀 다른 역할**이니 구분해서 보자

### "기준(어느 쪽을 다 살릴지)"을 정하는 건 오직 여기예요

sql

```sql
FROM EMP E LEFT OUTER JOIN DEPT D        
		   ↑  이 위치가"LEFT"(왼쪽)
```

**LEFT OUTER JOIN**에서 "LEFT(왼쪽)"이라는 건, **JOIN 키워드를 기준으로 왼쪽에 물리적으로 적혀 있는 테이블**을 말하는 거예요.

sql

```sql
FROM[여기가 왼쪽] LEFT OUTER JOIN [여기가 오른쪽]       
        EMP                          DEPT
```

→ 지금 문장에서는 EMP가 JOIN 키워드보다 **왼쪽에 써 있으니까** EMP가 "왼쪽 테이블" = 기준이 되는 거

### SELECT는 그냥 "결과에서 뭘 보여줄지" 고르는 것뿐

sql

```sql
SELECT E.EMPNO, E.ENAME, D.DNAME
```

→ 이건 이미 JOIN이 다 끝나서 만들어진 **합쳐진 결과표에서, 그중 몇 개 컬럼만 골라서 보여달라**는 뜻일 뿐

왼쪽 테이블의 모든 행과 오른쪽 테이블의 일치하는 행을 반환합니다. 일치하는 행이 없으면 NULL로 표시

SELECT E.EMPNO, E.ENAME, D.DNAME

FROM EMP E LEFT OUTER JOIN DEPT D

ON E.DEPTNO = D.DEPTNO;

오라클 문법

SELECT E.EMPNO, E.ENAME, D.DNAME

FROM EMP E, DEPT D

WHERE E.DEPTNO = D.DEPTNO(+);

**RIGHT OUTER JOIN**

오른쪽 테이블의 모든 행과 왼쪽 테이블의 일치하는 행을 반환합니다.

SELECT E.EMPNO, E.ENAME, D.DNAME

FROM EMP E RIGHT OUTER JOIN DEPT D

ON E.DEPTNO = D.DEPTNO;

**FULL OUTER JOIN**

양쪽 테이블의 모든 행을 반환합니다. 일치하지 않는 행은 NULL로 표시됩니다.

SELECT E.EMPNO, E.ENAME, D.DNAME

FROM EMP E FULL OUTER JOIN DEPT D

ON E.DEPTNO = D.DEPTNO;

**CROSS JOIN**

두 테이블 간의 모든 행을 조합합니다(카티시안 곱).

**두 집합의 모든 조합을 다 만드는 것**을 "카티시안 곱(Cartesian Product)

SELECT E.ENAME, D.DNAME

FROM EMP E CROSS JOIN DEPT D;

**실수 (ON 빼먹음)**

sql

```sql
SELECT E.ENAME, D.DNAME
FROM EMP E JOIN DEPT D;
```

→ **"어떻게 짝지어야 하는지" 아무 규칙이 없음.** 그러면 SQL은 "규칙이 없으니 그냥 다 조합할게"라고 판단하고 **CROSS JOIN처럼 동작**

### 예상 문제 스타일 1: 결과 행 수 계산형

```
[EMP 테이블] - 총 4행
[DEPT 테이블] - 총 3행

다음 SQL의 실행 결과 행 수는?

SELECT E.ENAME, D.DNAMEFROM EMP E, DEPT D;

① 3   ② 4   ③ 7   ④ 12
```

→ 이건 **콤마로 테이블만 나열하고 WHERE 조건(또는 ON)이 없는** 케이스예요. ON이 없으니 CROSS JOIN처럼 동작해서 **4 × 3 = 12행**이 정답이에요.

### 예상 문제 스타일 2: 개념 확인형

```
다음 중 CROSS JOIN에 대한 설명으로 옳은 것은?

① JOIN 조건절 없이 FROM절에 테이블을 나열하면 CROSS JOIN이 발생할 수 있다
② CROSS JOIN의 결과 행 수는 두 테이블의 행 수를 더한 값이다
③ CROSS JOIN은 항상 명시적으로 ON절을 필요로 한다
④ CROSS JOIN은 INNER JOIN보다 항상 결과 행 수가 적다
```

→ 정답은 ① (②는 "더한 값"이 아니라 "곱한 값"이라 틀림, ③은 CROSS JOIN 자체는 원래 ON절이 필요 없는 게 특징이라 틀림, ④는 정반대라 틀림)

### **5.4 등가 JOIN과 비등가 JOIN**

sql

```sql
SELECT E.ENAME, E.SAL, S.GRADE
FROM EMP E JOIN SALGRADE S
ON E.SAL BETWEEN S.LOSALAND S.HISAL;
```

**왜 `=`를 못 쓸까?** 를 이해하는 게 핵심이에요. 예를 들어 SALGRADE가 이렇다면:

|GRADE|LOSAL|HISAL|
|---|---|---|
|1|700|1200|
|2|1201|1400|

직원 SMITH의 SAL이 **800**이라고 해봐요. SALGRADE 테이블 어디에도 "800"이라는 숫자 자체는 없어요. 대신 800이 **700~1200 사이의 범위에 속하니까** GRADE 1로 매칭되는 거예요.

→ 만약 `ON E.SAL = S.LOSAL` 이렇게 등가로 짜버리면, SAL이 정확히 700이나 1201처럼 경계값이 아닌 이상 **아무것도 매칭이 안 돼서 결과가 텅 비어버려요.** 그래서 범위 비교(`BETWEEN`, `>`, `<`)가 꼭 필요한 케이스예요.

### 5.5 NATURAL JOIN — 시험에서 "위험성" 포인트로 자주 나옴

sql

```sql
SELECT ENAME, DNAMEFROM EMPNATURALJOIN DEPT;
```

**시험 함정**: "만약 EMP와 DEPT 두 테이블에 우연히 이름이 같은 컬럼이 DEPTNO 말고 하나 더 있다면?" (예: 둘 다 `CREATED_DATE`라는 컬럼을 갖고 있다면)

→ NATURAL JOIN은 **이름이 같은 컬럼을 전부 다** 자동으로 조인 조건에 넣어버려요. 그래서 의도치 않게 `DEPTNO`뿐 아니라 `CREATED_DATE`까지 같아야 매칭되는 상황이 생겨서, **결과가 예상보다 훨씬 적게 나오는 사고**가 날 수 있어요.

**USING**이 이 문제의 해결책으로 나오는 거예요. "여러 개 이름 같은 컬럼 중, 딱 이거 하나만 조인 조건으로 써라"라고 **명시적으로 지정**할 수 있으니까요.

sql

```sql
FROM EMP JOIN DEPT USING(DEPTNO)-- DEPTNO만 조인조건으로 쓰겠다고 못박음
```

### 5.6 3개 이상 JOIN — 실행 순서 감각이 중요

sql

```sql
FROM EMP E
JOIN DEPT D ON E.DEPTNO = D.DEPTNO 
JOIN LOCATIONS L ON D.LOC_ID = L.LOC_ID;
```

**핵심**: 이건 한 번에 3개가 동시에 합쳐지는 게 아니라, **순서대로 단계적으로** 합쳐짐.

```
1단계: EMP + DEPT 를 먼저 합쳐서 임시 결과표 생성
2단계: 그 임시 결과표 + LOCATIONS 를 다시 합침
```

## 3개 이상 테이블 JOIN — 예시 테이블로 확인

**EMP (사원)**

|EMPNO|ENAME|DEPTNO|
|---|---|---|
|1001|김철수|10|
|1002|이영희|20|
|1003|박민수|10|

**DEPT (부서)**

|DEPTNO|DNAME|LOC_ID|
|---|---|---|
|10|영업부|100|
|20|개발부|200|

**LOCATIONS (지역)**

|LOC_ID|CITY|
|---|---|
|100|서울|
|200|부산|

### 쿼리

sql

```sql
SELECT E.ENAME, D.DNAME, L.CITY
FROM EMP E JOIN DEPT D ON E.DEPTN O= D.DEPTNO 
JOIN LOCATIONS L ON D.LOC_ID= L.LOC_ID;
```

### 1단계: EMP + DEPT 먼저 합치기 (첫 번째 JOIN)

sql

```sql
FROM EMP E JOIN DEPT D ON E.DEPTNO = D.DEPTNO
```

**EMP의 DEPTNO와 DEPT의 DEPTNO가 같은 것끼리 매칭**

|EMPNO|ENAME|DEPTNO|DNAME|LOC_ID|
|---|---|---|---|---|
|1001|김철수|10|영업부|100|
|1002|이영희|20|개발부|200|
|1003|박민수|10|영업부|100|

→ **여기까지가 1단계 결과예요.** 이게 마치 "새로운 임시 테이블"처럼 만들어졌다고 생각. 이 임시 테이블 안에는 이제 **EMP 컬럼도 있고, DEPT 컬럼도 다 들어있어요** (LOC_ID까지 포함).

### 2단계: (1단계 결과) + LOCATIONS 합치기 (두 번째 JOIN)

sql

```sql
JOIN LOCATIONS L ON D.LOC_ID = L.LOC_ID
```

**여기서 핵심**: `D.LOC_ID`라고 썼지만, 사실 이건 **1단계에서 이미 만들어진 임시 테이블 안에 들어있는 LOC_ID 컬럼**을 가리키는 거예요. D라는 원래 테이블을 다시 조회하는 게 아니라요.

**1단계 결과의 LOC_ID**를 LOCATIONS의 LOC_ID와 매칭:

|EMPNO|ENAME|DEPTNO|DNAME|LOC_ID|CITY|
|---|---|---|---|---|---|
|1001|김철수|10|영업부|100|서울|
|1002|이영희|20|개발부|200|부산|
|1003|박민수|10|영업부|100|서울|

### 최종 SELECT 적용

sql

```sql
SELECT E.ENAME, D.DNAME, L.CITY
```

### 최종 결과

|ENAME|DNAME|CITY|
|---|---|---|
|김철수|영업부|서울|
|이영희|개발부|부산|
|박민수|영업부|서울|

### 그림으로 흐름 정리

```
[1단계]EMP ──(DEPTNO 기준)── DEPT        
								↓ 합쳐짐   
					[EMP+DEPT 임시결과]  ← 여기 안에 LOC_ID도 포함되어 있음!

[2단계]
[EMP+DEPT 임시결과] ──(LOC_ID 기준)── LOCATIONS        
				↓ 합쳐짐   [최종 결과: EMP+DEPT+LOCATIONS]
```

### 핵심 포인트

> **JOIN은 한 번에 3개를 동시에 합치는 게 아니라, "두 개씩 순서대로" 이어붙임.**
> 
> 1단계 결과(EMP+DEPT)가 마치 하나의 새 테이블처럼 되고, 2단계는 그 "새 테이블"과 LOCATIONS를 다시 합침.

**그래서 두 번째 JOIN의 ON절에서 `D.LOC_ID`를 쓸 수 있는 이유**는, D(DEPT)가 1단계 결과 안에 이미 녹아들어가 있어서, 그 안의 LOC_ID 컬럼을 계속 참조할 수 있기 때문이에요. E, D, L 별칭은 원래 테이블을 가리키는 이름표지만, JOIN이 진행될수록 그 이름표를 가진 컬럼들이 계속 "누적된 하나의 큰 표" 안에 같이 들어있다고 생각하면 됨.

**등가 JOIN (EQUI JOIN)**

= 연산자를 사용하여 두 테이블의 칼럼이 같은 값인 경우를 조인합니다.

SELECT E.ENAME, D.DNAME

FROM EMP E JOIN DEPT D

ON E.DEPTNO = D.DEPTNO;

**비등가 JOIN (NON-EQUI JOIN)**

= 외의 연산자(>, <, BETWEEN 등)를 사용하여 조인합니다.

SELECT E.ENAME, E.SAL, S.GRADE

FROM EMP E JOIN SALGRADE S

ON E.SAL BETWEEN S.LOSAL AND S.HISAL;

### **5.5 NATURAL JOIN과 USING 조건절**

**NATURAL JOIN**

NATURAL JOIN은 자동 조인이라 별칭을 안 붙이는 경우가 많음

두 테이블에 동일한 이름을 가진 모든 칼럼에 대해 자동으로 조인합니다.

SELECT ENAME, DNAME

FROM EMP NATURAL JOIN DEPT;

**USING 조건절**

조인할 특정 칼럼을 명시합니다. 동일한 이름의 칼럼이 하나 이상 있을 때 유용합니다.

SELECT ENAME, DNAME, DEPTNO

FROM EMP JOIN DEPT USING (DEPTNO);

### **5.6 3개 이상 테이블 JOIN**

SELECT E.ENAME, D.DNAME, L.CITY

FROM EMP E

JOIN DEPT D ON E.DEPTNO = D.DEPTNO

JOIN LOCATIONS L ON D.LOC_ID = L.LOC_ID;

### **5.7 실무 주의사항**

- JOIN 시 카디널리티(유일성)가 낮은 칼럼으로 조인하면 결과 행 수가 급격히 증가할 수 있습니다.
- 적절한 인덱스가 있는 칼럼으로 JOIN해야 성능이 좋습니다.
- 대용량 테이블 JOIN 시 실행 계획을 확인하고 최적화해야 합니다.

### **정리 및 핵심 포인트**

1. SELECT 문은 "**F**ROM → **W**HERE → **G**ROUP BY → **H**AVING → **S**ELECT → **O**RDER BY" 순서로 실행된다
2. 함수는 단일행/다중행으로 구분된다. 단일행 함수는 각 행별 결과, 집계 함수는 그룹 전체 결과를 반환한다.
3. WHERE 절은 그룹화 이전 필터, HAVING 절은 그룹화 이후 필터이다 집계 함수를 쓰려면 HAVING을 사용한다.
4. INNER JOIN은 교집합, OUTER JOIN은 포함 관계를 유지하며 NULL 로 채운다. 조인 조건에 인덱스가 잡히도록 설계하면 성능을 크게 개선할 수 있다.
5. 동적 쿼리를 자주 작성한다면 WHERE 1=1 패턴으로 조건을 손쉽게 추가/제거하고, 별칭·CASE·서브쿼리를 적절히 사용해 가독성과 재사용성을 높인다.
6. SQL은 단순 질의어를 넘어 데이터 통합·보고·품질 관리 등 거의 모든 데이터 작업의 공통 언어다. 핵심 문법과 패턴을 체득하면 현업 데이터베이스뿐 아니라 BI ETL·데이터 분석 도구에서도 즉시 활용할 수 있다.

## **1. 서브쿼리**

### **서브쿼리 개요**

서브쿼리의 기초 개념

쿼리(Query)는 하나의 명령 문장을 의미하며, 이를 메인쿼리(Mainquery)라고 합니다. 그 명령 문장 안에 또 다른 쿼리가 있는데, 이를 서브쿼리(Subquery) 또는 부질의라고 합니다.

서브쿼리는 메인쿼리 내부에 괄호로 감싸진 형태로 존재하며, 다양한 절(WHERE, FROM, SELECT 등)에서 사용될 수 있습니다.

서브쿼리 사용 시점

- 복잡한 조건 필터링
- 계산된 칼럼 생성
- 임시 테이블(INLINE-VIEW) 활용
- 조인보다 간단한 존재 여부 판정(예: EXISTS, IN)

### **서브쿼리 종류**

동작 방식에 따른 분류

**1. 연관 서브쿼리(Correlated Subquery)**

- 메인 쿼리의 각 행과 관련되어 처리
- 메인 쿼리 한 행을 읽을 때마다 서브 쿼리가 실행됨

SELECT ENAME, DEPTNO, SAL

FROM EMP E1

WHERE SAL > (SELECT AVG(SAL)

FROM EMP E2

WHERE E1.DEPTNO = E2.DEPTNO);

**2. 비연관 서브쿼리(Un-Correlated Subquery)**

- 메인 쿼리와 독립적으로 실행
- 한 번만 실행되어 결과를 메인 쿼리에 전달

SELECT ENAME, SAL

FROM EMP

WHERE SAL > (SELECT AVG(SAL) FROM EMP);

반환 형태에 따른 분류

**1. 단일행 서브쿼리(Single Row Subquery)**

- 하나의 행만 반환
- 단일행 비교 연산자(=, >, <, >=, <=, <>) 사용

SELECT ENAME, SAL

FROM EMP

WHERE SAL > (SELECT AVG(SAL) FROM EMP);

**2. 다중행 서브쿼리(Multiple Row Subquery)**

- 여러 행을 반환
- 다중행 비교 연산자(IN, ANY, ALL, EXISTS) 사용

SELECT ENAME, SAL

FROM EMP

WHERE DEPTNO IN (SELECT DEPTNO FROM DEPT WHERE LOC = 'NEW YORK');

**3. 다중칼럼 서브쿼리(Multiple Column Subquery)**

- 여러 칼럼을 반환
- 복합 조건 비교에 사용

SELECT *

FROM EMP

WHERE (JOB, DEPTNO) IN (SELECT JOB, DEPTNO FROM EMP WHERE ENAME = 'SMITH');

### **서브쿼리 활용법**

WHERE 절에서의 서브쿼리

SELECT ENAME, SAL

FROM EMP

WHERE SAL > (SELECT AVG(SAL) FROM EMP)

ORDER BY SAL DESC;

HAVING 절에서의 서브쿼리

SELECT DEPTNO, AVG(SAL)

FROM EMP

GROUP BY DEPTNO

HAVING AVG(SAL) > (SELECT AVG(SAL) FROM EMP);

SELECT 절에서의 서브쿼리 (스칼라 서브쿼리)

SELECT ENAME, SAL,

(SELECT AVG(SAL) FROM EMP) AS AVG_SAL

FROM EMP

ORDER BY SAL DESC;

FROM 절에서의 서브쿼리(인라인 뷰)

SELECT E.ENAME, E.SAL, A.AVG_SAL

FROM EMP E,

(SELECT AVG(SAL) AS AVG_SAL FROM EMP) A

WHERE E.SAL > A.AVG_SAL

ORDER BY SAL DESC;

DML에서의 서브쿼리 활용

**1. INSERT와 서브쿼리**

INSERT INTO EMP (EMPNO, ENAME, SAL)

VALUES ((SELECT MAX(EMPNO) + 1 FROM EMP), 'NEW_EMP', 3000);

**2. UPDATE와 서브쿼리**

UPDATE EMP

SET SAL=SAL*1.1

WHERE DEPTNO = (SELECT DEPTNO FROM DEPT WHERE DNAME = 'SALES');

**3. DELETE와 서브쿼리**

DELETE FROM EMP

WHERE SAL < (SELECT AVG(SAL) FROM EMP);

### **뷰(VIEW)의 개념과 활용법**

뷰의 개념

뷰는 하나 이상의 테이블에 있는 데이터의 부분집합을 논리적으로 표현한 가상 테이블입니다. 실제 데이터를 가지고 있지 않고, 데이터를 추출하는 SQL문을 저장하고 있습니다.

뷰의 장점

- 복잡한 쿼리 단순화
- 데이터 보안 강화(필요한 칼럼만 제공)
- 데이터 독립성 확보
- 동일 데이터에 대한 다양한 관점 제공

뷰의 생성과 활용

CREATE VIEW EMP_DEPT_V AS

SELECT E.EMPNO, E.ENAME, D.DNAME, E.SAL

FROM EMP E, DEPT D

WHERE E.DEPTNO = D.DEPTNO;

- - 뷰 사용

SELECT * FROM EMP_DEPT_V WHERE SAL > 2000;

## **2. 집합 연산자**

### **집합 연산자 개념**

집합 연산자는 두 개 이상의 SELECT 문의 결과를 하나의 결과로 결합하는 연산자입니다. 특히 회계나 통계 분석 등에서 유용하게 사용됩니다.

### **집합 연산자 종류**

**UNION (합집합, 중복 제거)**

두 개의 SELECT 문 결과를 하나로 합치고, 중복된 행은 제거합니다.

SELECT DEPTNO, DNAME FROM DEPT

UNION

SELECT DEPTNO, JOB FROM EMP;

**UNION ALL (합집합, 중복 포함)**

두 개의 SELECT 문 결과를 하나로 합치고, 중복된 행도 그대로 포함합니다.

SELECT DEPTNO FROM DEPT

UNION ALL

SELECT DEPTNO FROM EMP;

**INTERSECT (교집합)**

두 SELECT 문의 결과에 모두 포함된 행만 반환합니다.

SELECT DEPTNO FROM DEPT

INTERSECT

SELECT DEPTNO FROM EMP;

**MINUS / EXCEPT (차집합)**

첫 번째 SELECT 문의 결과에서 두 번째 SELECT 문의 결과에 포함된 행을 제외한 나머지를 반환합니다.

(Oracle은 MINUS, SQL Server는 EXCEPT 사용)

SELECT DEPTNO FROM DEPT

MINUS

SELECT DEPTNO FROM EMP;

집합 연산자 사용 조건

- 각 SELECT 문의 칼럼 수가 동일해야 합니다.
- 대응되는 칼럼의 데이터 타입이 같거나 호환 가능해야 합니다.
- 결과는 첫 번째 SELECT 문의 칼럼 이름을 따릅니다.
- ORDER BY 절은 전체 결과에 대해 마지막에 한 번만 사용합니다.

## **3. 그룹 함수**

### **그룹 함수 개념**

그룹 함수는 데이터를 다차원적으로 집계하는 기능을 제공합니다. GROUP BY 절과 함께 사용하여 그룹별 합계, 소계, 총계 등의 다양한 통계 정보를 산출할 수 있습니다.

### **다차원 집계**

**ROLLUP**

ROLLUP은 GROUP BY 절과 함께 사용하여 지정된 칼럼의 소계와 총계를 자동으로 산출합니다. 계층적 구조로 집계를 생성합니다.

SELECT DEPTNO, JOB, SUM(SAL) AS TOTAL_SALARY

FROM EMP

GROUP BY ROLLUP(DEPTNO, JOB)

ORDER BY DEPTNO;

이 쿼리는 다음 그룹별 총합을 계산합니다:

- 각 부서와 직무 조합
- 각 부서별
- 전체 총합

**CUBE**

CUBE는 ROLLUP과 유사하지만, 지정된 칼럼들의 가능한 모든 조합에 대해 집계를 수행합니다.

SELECT DEPTNO, JOB, SUM(SAL)

FROM EMP

GROUP BY CUBE(DEPTNO, JOB);

이 쿼리는 다음 그룹별 총합을 계산합니다:

- 각 부서와 직무 조합
- 각 부서별
- 각 직무별
- 전체 총합

**GROUPING SETS**

GROUPING SETS는 원하는 특정 그룹의 집계만 수행할 수 있도록 해줍니다. ROLLUP이나 CUBE의 일부 기능만 필요할 때 유용합니다.

SELECT DEPTNO, JOB, SUM(SAL)

FROM EMP

GROUP BY GROUPING SETS ((DEPTNO, JOB), (DEPTNO), (JOB), ());

이 쿼리는 다음 그룹별 총합을 계산합니다:

- 각 부서와 직무 조합
- 각 부서별
- 각 직무별
- 전체 총합

### **GROUPING/CASE/DECODE 활용**

**GROUPING 함수**

GROUPING 함수는 특정 칼럼이 집계에 사용되었는지 여부를 반환합니다. 집계에 사용되었으면 0, 그렇지 않으면 1을 반환합니다.

SELECT

CASE GROUPING(DEPTNO) WHEN 1 THEN '전체 부서' ELSE TO_CHAR(DEPTNO) END AS DEPTNO,

CASE GROUPING(JOB) WHEN 1 THEN '전체 직무' ELSE JOB END AS JOB,

SUM(SAL) AS TOTAL_SALARY

FROM EMP

GROUP BY ROLLUP(DEPTNO, JOB);

**CASE와 DECODE 활용**

CASE 표현식이나 DECODE 함수를 사용하여 집계 결과의 가독성을 높일 수 있습니다.

Oracle DECODE 예시

SELECT

DECODE(GROUPING(DEPTNO), 1, '전체 부서', TO_CHAR(DEPTNO)) AS DEPTNO,

DECODE(GROUPING(JOB), 1, '전체 직무', JOB) AS JOB,

SUM(SAL) AS TOTAL_SALARY

FROM EMP

GROUP BY ROLLUP(DEPTNO, JOB);

CASE 문 예시

SELECT

CASE

WHEN GROUPING(DEPTNO) = 1 THEN '전체 부서'

ELSE TO_CHAR(DEPTNO)

END AS DEPTNO,

CASE

WHEN GROUPING(JOB) = 1 THEN '전체 직무'

ELSE JOB

END AS JOB,

SUM(SAL) AS TOTAL_SALARY

FROM EMP

GROUP BY ROLLUP(DEPTNO, JOB);

## **4. 윈도우 함수**

### **윈도우 함수 개념**

윈도우 함수는 행과 행 간의 관계를 쉽게 정의하기 위해 제공되는 함수입니다. 특정 윈도우(파티션)에 대해 계산을 수행합니다. 집계 함수와 달리 행이 그대로 유지되면서 값이 추가됩니다.

### **윈도우 함수 기본 구문**

윈도우함수(인자) OVER (

[PARTITION BY 칼럼]

[ORDER BY 칼럼]

[ROWS | RANGE BETWEEN 시작점 AND 끝점]

)

- PARTITION BY: 계산을 적용할 행 그룹 지정
- ORDER BY: 파티션 내에서 행 순서 지정
- ROWS | RANGE: 프레임 지정(현재 행을 기준으로 계산할 범위)

### **윈도우 함수 종류**

순위 관련 함수

**1. ROW_NUMBER()**

- 파티션 내에서 순차적인 번호 부여(중복 값에도 순차적인 번호)

SELECT ENAME, SAL,

ROW_NUMBER() OVER(ORDER BY SAL DESC) AS RN

FROM EMP;

**2. RANK()**

- 파티션 내에서 순위를 매기되, 동일한 값은 동일한 순위(다음 순위는 건너뜀)

SELECT ENAME, SAL,

RANK() OVER(ORDER BY SAL DESC) AS RANK_SAL

FROM EMP;

**3. DENSE_RANK()**

- RANK와 유사하나, 동일한 순위 다음에 순차적인 값을 부여(건너뛰지 않음)

SELECT ENAME, SAL,

DENSE_RANK() OVER(ORDER BY SAL DESC) AS DENSE_RANK_SAL

FROM EMP;

집계 관련 함수

**1. SUM() OVER**

- 파티션별 합계 계산

SELECT ENAME, DEPTNO, SAL,

SUM(SAL) OVER(PARTITION BY DEPTNO) AS DEPT_SUM

FROM EMP;

**2. AVG() OVER**

- 파티션별 평균 계산

SELECT ENAME, DEPTNO, SAL,

AVG(SAL) OVER(PARTITION BY DEPTNO) AS DEPT_AVG

FROM EMP;

**3. MAX() OVER, MIN() OVER**

- 파티션별 최댓값, 최솟값 계산

SELECT ENAME, DEPTNO, SAL,

MAX(SAL) OVER(PARTITION BY DEPTNO) AS DEPT_MAX,

MIN(SAL) OVER(PARTITION BY DEPTNO) AS DEPT_MIN

FROM EMP;

행 이동 관련 함수

**1. LAG()**

- 이전 행의 값을 참조

SELECT ENAME, HIREDATE, SAL,

LAG(SAL) OVER(ORDER BY HIREDATE) AS PREV_SAL

FROM EMP;

**2. LEAD()**

- 다음 행의 값을 참조

SELECT ENAME, HIREDATE, SAL,

LEAD(SAL) OVER(ORDER BY HIREDATE) AS NEXT_SAL

FROM EMP;

누적 관련 함수

**1. CUME_DIST()**

- 누적 분포 값을 계산(현재 행보다 작거나 같은 값의 누적 개수 / 전체 행 수)

SELECT ENAME, SAL,

CUME_DIST() OVER(ORDER BY SAL) AS CUM_DIST

FROM EMP;

**2. PERCENT_RANK()**

- 백분위 순위 계산((현재 행 순위 - 1) / (전체 행 수 - 1))

SELECT ENAME, SAL,

PERCENT_RANK() OVER(ORDER BY SAL) AS PCT_RANK

FROM EMP;

## **5. Top N 쿼리**

### **ROWNUM 슈도 칼럼**

ROWNUM은 Oracle에서 제공하는 가상 칼럼(pseudo column)으로 쿼리 결과의 각 행에 1부터 시작하는 순차적인 번호를 부여합니다.

SELECT ROWNUM, ENAME, SAL

FROM EMP

WHERE ROWNUM <= 5;

### **Top N 쿼리 활용법**

Oracle에서의 Top N 쿼리(ROWNUM 활용)

- - 잘못된 방법 (원하는 결과가 나오지 않음)

SELECT ROWNUM, ENAME, SAL

FROM EMP

WHERE ROWNUM <= 5

ORDER BY SAL DESC;

- - 올바른 방법 (인라인 뷰 활용)

SELECT ROWNUM, E.*

FROM (SELECT ENAME, SAL

FROM EMP

ORDER BY SAL DESC) E

WHERE ROWNUM <= 5;

SQL Server에서의 Top N 쿼리

- - SQL Server의 TOP 사용

SELECT TOP 5 ENAME, SAL

FROM EMP

ORDER BY SAL DESC;

표준 SQL의 FETCH 절 사용

- - 표준 SQL의 FETCH 문 사용(Oracle 12c 이상, SQL Server 2012 이상)

SELECT ENAME, SAL

FROM EMP

ORDER BY SAL DESC

FETCH FIRST 5 ROWS ONLY;

페이징 처리

- - 11~20번째 행 조회

SELECT *

FROM (

SELECT ROWNUM AS RN, E.*

FROM (SELECT * FROM EMP ORDER BY SAL DESC) E

WHERE ROWNUM <= 20

)

WHERE RN >= 11;

## **6. 셀프 조인과 계층형 질의**

### **셀프 조인 개념 및 활용**

셀프 조인(Self Join)은 하나의 테이블을 마치 두 개의 테이블인 것처럼 조인하는 기법입니다. 주로 계층적 관계나 순차적 관계를 조회할 때 사용됩니다.

- - 사원과 관리자 정보 함께 조회

SELECT E.EMPNO, E.ENAME AS EMP_NAME,

M.EMPNO AS MGR_EMPNO, M.ENAME AS MGR_NAME

FROM EMP E, EMP M

WHERE E.MGR = M.EMPNO;

- - 또는 JOIN 문법 사용

SELECT E.EMPNO, E.ENAME AS EMP_NAME,

M.EMPNO AS MGR_EMPNO, M.ENAME AS MGR_NAME

FROM EMP E LEFT JOIN EMP M

ON E.MGR = M.EMPNO;

### **계층형 질의 개념 및 문법**

계층형 질의는 테이블에 저장된 계층적 구조의 데이터를 조회하는 방법입니다. Oracle에서는 START WITH와 CONNECT BY 절을 사용합니다.

- - 기본 구문

SELECT [컬럼 목록], LEVEL

FROM [테이블]

START WITH [최상위 조건]

CONNECT BY [PRIOR 부모키 = 자식키 | 자식키 = PRIOR 부모키];

계층형 질의 주요 요소

- START WITH: 계층 구조의 시작점(루트 노드) 지정
- CONNECT BY: 부모-자식 관계를 정의
- LEVEL: 계층의 깊이를 나타내는 의사 컬럼
- CONNECT_BY_ROOT: 현재 행의 최상위 조상 노드를 반환
- SYS_CONNECT_BY_PATH: 루트부터 현재 노드까지의 경로를 문자열로 반환
- CONNECT_BY_ISLEAF: 현재 행이 리프 노드인지 여부(1 또는 0)

계층형 질의 예제

- - 조직도 조회(최상위 관리자부터)

SELECT ENAME, EMPNO, MGR, LEVEL,

LPAD(' ', (LEVEL-1)*2) || ENAME AS ORG_CHART

FROM EMP

START WITH MGR IS NULL

CONNECT BY PRIOR EMPNO = MGR

ORDER SIBLINGS BY ENAME;

## **7. PIVOT 절과 UNPIVOT 절**

### **PIVOT 절 개념 및 문법**

PIVOT은 행을 열로 변환해주는 기능을 수행합니다. 특히 보고서 형태의 출력이나 교차 테이블을 생성할 때 유용합니다.

- - 기본 구문

SELECT *

FROM (기준 테이블)

PIVOT (

집계함수(집계대상컬럼)

FOR 피벗컬럼 IN (피벗값1, 피벗값2,...)

);

PIVOT 활용 예제

- - 부서별, 직무별 사원 수 피벗 테이블

SELECT *

FROM (SELECT DEPTNO, JOB FROM EMP)

PIVOT (

COUNT(*)

FOR JOB IN ('CLERK' AS CLERK, 'MANAGER' AS MANAGER, 'ANALYST' AS ANALYST,

'SALESMAN' AS SALESMAN)

)

ORDER BY DEPTNO;

### **UNPIVOT 절 개념 및 문법**

UNPIVOT은 PIVOT의 반대 기능으로, 열을 행으로 변환합니다.

- - 기본 구문

SELECT *

FROM (피벗된 테이블)

UNPIVOT (

값을_저장할_컬럼이름

FOR 컬럼을_저장할_컬럼이름 IN (컬럼1, 컬럼2, ...)

);

UNPIVOT 활용 예제

- - 피벗된 테이블을 다시 원래 형태로 변환

SELECT *

FROM (피벗된 테이블 또는 서브쿼리)

UNPIVOT (

EMP_COUNT

FOR JOB_TYPE IN (CLERK, MANAGER, ANALYST, SALESMAN)

);

## **8. 정규표현식**

### **정규표현식 개념**

정규표현식은 문자열 내에서 특정 패턴을 검색, 추출, 변환하는 데 사용되는 강력한 도구입니다.

### **정규표현식 연산자**

| 연산자 | 설명 | 예시  
|  
| --- | --- | --- |  
| . | 임의의 단일 문자 | 'a.c'는 'abc', 'adc' 등과 일치 |  
| | | OR 연산자 | 'a|b'는 'a' 또는 'b'와 일치 |  
| * | 앞 요소가 0회 이상 반복 | 'ab*'는 'a', 'ab', 'abb' 등과 일치 |  
| + | 앞 요소가 1회 이상 반복 | 'ab+'는 'ab', 'abb' 등과 일치(단, 'a'는 미포함) |  
| ? | 앞 요소가 0회 또는 1회 반복 | 'ab?'는 'a', 'ab'와 일치 |  
| {n} | 앞 요소가 정확히 n회 반복 | 'ab{3}'는 'abbb'와 일치 |  
| {n,} | 앞 요소가 n회 이상 반복 | |

### **정리 및 핵심 포인트**

1. 서브쿼리 위치(SELECT FROM-WHERE HAVING)마다 역할이 다르다 - 인라인 뷰는 복잡한 조인 대신 임시 테이블처럼 활용하면 가독성이 크게 향상된다.
2. 집합 연산자는 칼럼 수·데이터형이 일치해야 하며, UNION ALL이 UNION보다 빠르지만 중복을 제거하지 않는다. 마지막 ORDER BY는 전체 결과에 한 번만 작성한다.
3. ROLLUP은 계층적 합계, CUBE는 모든 조합 합계, GROUPING SETS는 선택적 합계를 만든다. GROUPING 함수로 '총계' 표시를 제어하면 보고서 품질이 높아진다.
4. 윈도우 함수는 행을 지우지 않고 통계열을 추가한다. ORDER BY만 있고 PARTITION BY가 없으면 전 행을 한 그룹으로 본다-프레임 범위도 성능에 영향.
5. Top N 쿼리에서 ROWNUM은 정렬 후에 매겨야 정확하다. 표준 SQL: FETCH FIRST n ROWS ONLY를 기억해 두면 DBMS 이식성을 확보할 수 있다.
6. 셀프 조인은 자체 테이블의 부모·자식 관계나 시간대별 이전 상태 비교에 필수적이다. 계층형 질의에서는 LEVEL, SYS_CONNECT_BY_PATH 등 의사 컬럼이 유용하다.
7. PIVOT/UNPIVOT은 집계 함수가 필수이며, 열 이름은 IN 절에 미리 명시하는 것이 안전하다. 정규표현식은 오라클의 REGEXP_LIKE-REPLACE SUBSTR 등과 함께 쓰면 데이터 정제 속도가 획기적으로 빨라진다.

# **2과목 SQL 기본과 활용 - 3장 관리구분**

## **학습목표**

- SQL 명령 분류(DML・TCL・DDL DCL)를 구분하고, 각 유형이 데이터·트랜잭션 스키마·권한에 미치는 역할을 설명할 수 있다.
- INSERT-UPDATE-DELETE-MERGE 등 DML 문을 사용해 데이터를 삽입·수정·삭제·동기화할 수 있다.
- COMMIT-ROLLBACK-SAVEPOINT를 포함한 TCL 명령으로 트랜잭션을 제어하고, ACID 특성을 검증·적용할 수 있다.
- CREATE ALTER-DROP TRUNCATE 등 DDL 명령어로 테이블·제약조건 인덱스 등을 정의·변경·삭제할 수 있다.
- DCL(GRANT REVOKE ROLE)을 이용해 사용자 권한을 효율적으로 관리하고, 보안 정책을 수립할 수 있다.

## **목차**

1. DML (Data Manipulation Language, 데이터 조작어)
2. TCL (Transaction Control Language, 트랜잭션 제어어)
3. DDL (Data Definition Language, 데이터 정의어)
4. DCL (Data Control Language, 데이터 제어어)

## **1. DML (데이터 조작어)**

DML은 데이터베이스의 테이블에 데이터를 추가, 수정, 삭제하는 명령어입니다. 주요 명령어로는 INSERT, UPDATE, DELETE, MERGE가 있습니다.

### **1.1 INSERT 문**

INSERT 문은 데이터베이스 테이블에 새로운 데이터를 추가할 때 사용하는 SQL 명령어입니다.

**1.1.1 단일행 INSERT 문**

기본 형식

INSERT INTO 테이블명 (칼럼1, 칼럼2, ...)

VALUES (값1, 값2, ...);

모든 칼럼에 값을 삽입할 경우 칼럼 이름을 생략할 수 있습니다.

INSERT INTO 테이블명

VALUES (값1, 값2, ...);

예시

- - 특정 칼럼만 지정하여 삽입

INSERT INTO EMP (EMPNO, ENAME, JOB, HIREDATE)

VALUES (101, 'YONGMOON', 'MANAGER', '2023-08-01');

- - 모든 칼럼 값 지정하여 삽입

INSERT INTO EMP

VALUES (102, 'YONGMOON2', 'CLERK', '7902', '2023-08-03', 2000, NULL, 20);

**1.1.2 서브쿼리를 이용한 다중행 INSERT 문**

다른 테이블에서 데이터를 추출해 새로운 테이블에 삽입할 때 유용합니다.

예시

- - EMP_MNG 테이블 생성

CREATE TABLE EMP_MNG (

EMPNO NUMBER(4,0),

ENAME VARCHAR2(10),

JOB VARCHAR2(9),

MGR NUMBER(4,0),

HIREDATE DATE,

SAL NUMBER(7,2),

COMM NUMBER(7,2),

DEPTNO NUMBER(2,0)

);

- - 서브쿼리를 이용한 다중행 INSERT

INSERT INTO EMP_MNG (

EMPNO,

ENAME,

JOB,

HIREDATE,

DEPTNO

)

SELECT

EMPNO,

ENAME,

JOB,

HIREDATE,

DEPTNO

FROM

EMP

WHERE

DEPTNO = 20;

서브쿼리를 이용하면 특정 조건에 맞는 다수의 데이터를 한 번에 다른 테이블로 삽입할 수 있습니다. 위 예시에서는 부서 번호가 20인 사원 정보만 선택적으로 삽입했습니다.

### **1.2 UPDATE 문**

UPDATE 문은 테이블에 이미 존재하는 데이터를 수정할 때 사용합니다.

기본 형식

UPDATE 테이블명

SET 칼럼1 = 값1, 칼럼2 = 값2, ...

WHERE 조건;

예시

- - 여러 칼럼을 동시에 UPDATE

UPDATE EMP

SET SAL = 2000, COMM = 300

WHERE ENAME = 'YONGMOON';

- - 서브쿼리를 이용한 UPDATE

UPDATE EMP

SET SAL = (SELECT AVG(SAL) FROM EMP WHERE DEPTNO = 10)

WHERE ENAME = 'YONGMOON';

두 번째 예시는 서브쿼리를 사용하여 DEPTNO가 10인 부서의 평균 급여를 계산하고, 그 값을 사용하여 YONGMOON의 SAL 값을 업데이트합니다.

### **1.3 DELETE 문**

DELETE 문은 테이블의 특정 행을 삭제할 때 사용합니다.

기본 형식

DELETE FROM 테이블명

WHERE 조건;

예시

DELETE FROM EMP

WHERE ENAME = 'YONGMOON2';

DELETE와 TRUNCATE의 차이점

| 특성 | DELETE | TRUNCATE  
|  
| --- | --- | --- |  
| 작동 방식 | 행 단위로 데이터를 삭제 | 테이블 구조를 유지한 채 모든 데이터를 한 번에 제거 |  
| 속도 | 각 행을 개별적으로 삭제하므로 상대적으로 느림 | 전체 테이블을 한 번에 비우므로 일반적으로 더 빠름 |  
| 트랜잭션 로그 | 각 삭제 작업을 트랜잭션 로그에 기록 | 최소한의 트랜잭션 로그만 생성 |  
| 롤백 가능성 | 트랜잭션 내에서 롤백이 가능 | 일반적으로 롤백이 불가능 |  
| WHERE 절 사용 | WHERE 절을 사용하여 특정 조건의 데이터만 삭제할 수 있음 | WHERE 절을 사용할 수 없으며, 항상 전체 테이블을 비움 |  
| 인덱스 영향 | 인덱스를 유지 | 인덱스를 재설정 |

### **1.4 MERGE 문**

MERGE 문은 데이터의 삽입, 업데이트, 삭제 작업을 하나의 명령문으로 처리할 수 있도록 도와주는 명령어입니다. 주로 소스 테이블의 데이터를 타깃 테이블에 반영할 때 사용됩니다.

기본 형식

MERGE INTO 타깃_테이블 타깃(별칭)

USING 소스_테이블 소스(별칭)

ON (타깃.조건_칼럼 = 소스.조건_칼럼)

WHEN MATCHED THEN

UPDATE SET 타깃.칼럼1 = 소스.칼럼1, ...

WHEN NOT MATCHED THEN

INSERT (칼럼1, 칼럼2, ...)

VALUES (소스.칼럼1, 소스.칼럼2, ...);

예시

MERGE INTO EMP_MNG M

USING EMP E

ON (M.EMPNO = E.EMPNO)

WHEN MATCHED THEN

UPDATE SET

M.SAL = E.SAL,

M.COMM = E.COMM

WHEN NOT MATCHED THEN

INSERT (EMPNO, ENAME, JOB, MGR, HIREDATE, SAL, COMM, DEPTNO)

VALUES (E.EMPNO, E.ENAME, E.JOB, E.MGR, E.HIREDATE, E.SAL, E.COMM, E.DEPTNO);

이 예시에서는 EMP_MNG와 EMP 테이블에서 EMPNO가 일치하는 행이 있을 경우, EMP 테이블의 SAL 값을 EMP_MNG 테이블의 SAL과 COMM 칼럼에 UPDATE합니다. 일치하지 않는 EMP 테이블의 행들은 EMP_MNG 테이블에 새로운 행으로 INSERT됩니다.

## **2. TCL (트랜잭션 제어어)**

TCL은 여러 DML 명령어로 구성된 트랜잭션을 관리하는 명령어입니다. 주요 명령어로는 COMMIT, ROLLBACK, SAVEPOINT가 있습니다.

### **2.1 트랜잭션의 기본 개념**

트랜잭션이란 데이터베이스 내에서 실행되는 하나의 논리적 작업 단위를 말하며, 여러 개의 연산을 포함할 수 있습니다.

트랜잭션의 ACID 속성

- 원자성(Atomicity): 모든 연산이 성공적으로 완료되거나 아무것도 실행되지 않은 상태를 유지
- 일관성(Consistency): 트랜잭션은 데이터베이스의 일관성을 유지
- 고립성(Isolation): 다른 트랜잭션의 영향을 받지 않고 독립적으로 실행
- 지속성(Durability): 성공적으로 수행된 트랜잭션의 결과는 시스템 오류가 발생해도 유지

### **2.2 COMMIT**

COMMIT을 실행하면 트랜잭션이 종료되고, 해당 트랜잭션에서 이루어진 모든 변경 사항이 데이터베이스에 영구적으로 저장됩니다.

예시

- - 1. 직원 삭제

DELETE FROM EMP WHERE ENAME = 'YONGMOON' OR ENAME = 'YONGMOON2';

- - 2. 새로운 직원 추가

INSERT INTO EMP (EMPNO, ENAME, JOB, MGR, HIREDATE, SAL, COMM, DEPTNO)

VALUES (7935, 'JAMESBOND', 'ANALYST', 7566, TO_DATE('2023-08-19', 'YYYY-MM-DD'), 3500, NULL, 20);

- - 3. 급여 10% 인상

UPDATE EMP SET SAL=SAL*1.1 WHERE EMPNO = 7935;

- - 4. 부서 번호 변경

UPDATE EMP SET DEPTNO = 30 WHERE EMPNO = 7902;

- - 5. 모든 변경 사항 확정

COMMIT;

COMMIT 이후에는 해당 트랜잭션에 대한 ROLLBACK이 불가능합니다.

### **2.3 ROLLBACK**

ROLLBACK은 트랜잭션의 모든 변경 사항을 취소하고, 트랜잭션 시작 시점으로 되돌리는 명령어입니다.

예시

- - 1. JAMESBOND (7935) 급여 40% 인상

UPDATE EMP SET SAL=SAL*1.4 WHERE EMPNO = 7935;

- - 2. JAMESBOND 부서 번호를 30으로 수정

UPDATE EMP SET DEPTNO = 30 WHERE EMPNO = 7935;

- - 3. 변경사항 전부 취소

ROLLBACK;

ROLLBACK 실행 후에는 트랜잭션 시작 전 상태로 되돌아갑니다.

### **2.4 SAVEPOINT**

SAVEPOINT는 트랜잭션 내에서 특정 지점을 지정하고, 후에 해당 지점까지만 롤백할 수 있게 하는 명령어입니다.

예시

- - 트랜잭션 시작

BEGIN;

- - 1. 첫 번째 INSERT

INSERT INTO EMP (EMPNO, ENAME, JOB, SAL, DEPTNO)

VALUES (8001, 'AAAAA', 'CLERK', 1500, 10);

- - 2. SAVEPOINT sp1

SAVEPOINT sp1;

- - 3. 두 번째 INSERT

INSERT INTO EMP (EMPNO, ENAME, JOB, SAL, DEPTNO)

VALUES (8002, 'BBBBB', 'MANAGER', 2500, 20);

- - 4. SAVEPOINT sp2

SAVEPOINT sp2;

- - 5. 세 번째 INSERT

INSERT INTO EMP (EMPNO, ENAME, JOB, SAL, DEPTNO)

VALUES (8003, 'CCCCC', 'ANALYST', 3000, 30);

- - 6. 오류로 sp2로 롤백

ROLLBACK TO sp2;

- - 7. 나머지 작업 확정

COMMIT;

위 예시에서 세 번째 INSERT는 취소되지만, 첫 번째와 두 번째 INSERT는 커밋됩니다. COMMIT 후에는 SAVEPOINT가 모두 무효화됩니다.

## **3. DDL (데이터 정의어)**

DDL은 데이터베이스에서 테이블, 인덱스, 뷰, 스키마 등 데이터베이스 객체를 정의하고 관리하기 위한 SQL 명령어들의 집합입니다.

### **3.1 CREATE TABLE**

CREATE TABLE 명령은 새로운 테이블을 생성하는 SQL 명령어입니다.

기본 형식

CREATE TABLE 테이블명 (

칼럼1 데이터타입 [제약조건],

칼럼2 데이터타입 [제약조건],

...

);

예시

CREATE TABLE STUDENT (

STUDENT_ID NUMBER(5) PRIMARY KEY,

NAME VARCHAR2(50) NOT NULL,

AGE NUMBER(3),

ENROLL_DATE DATE DEFAULT SYSDATE,

DEPT_ID NUMBER(3) REFERENCES DEPARTMENT(DEPT_ID)

);

참조 무결성 옵션

외래 키를 사용할 때 참조 무결성을 유지하기 위한 옵션을 지정할 수 있습니다:

- ON DELETE CASCADE: 부모 테이블의 행이 삭제되면 자식 테이블의 참조하는 행도 함께 삭제
- ON DELETE SET NULL: 부모 테이블의 행이 삭제되면 자식 테이블의 참조하는 열을 NULL로 설정
- ON DELETE SET DEFAULT: 부모 테이블의 행이 삭제되면 자식 테이블의 참조하는 열을 기본값으로 설정
- ON DELETE NO ACTION: 부모 테이블의 행을 삭제하려고 할 때 자식 테이블에 참조하는 행이 있으면 삭제 불가
- ON DELETE RESTRICT: NO ACTION과 유사하게 동작

예시

CREATE TABLE CHILD_TABLE_CASCADE (

CHILD_ID NUMBER PRIMARY KEY,

PARENT_ID NUMBER,

CONSTRAINT FK_PARENT_CASCADE FOREIGN KEY (PARENT_ID)

REFERENCES PARENT_TABLE (PARENT_ID) ON DELETE CASCADE

);

### **3.2 ALTER TABLE**

ALTER TABLE 명령어는 기존 테이블의 구조를 수정할 때 사용하는 SQL 명령어입니다.

주요 기능

- 칼럼 추가: ALTER TABLE 테이블명 ADD 칼럼명 데이터타입 [제약조건]
- 칼럼 삭제: ALTER TABLE 테이블명 DROP COLUMN 칼럼명
- 칼럼명 변경: ALTER TABLE 테이블명 RENAME COLUMN 기존칼럼명 TO 새칼럼명
- 데이터 타입 변경: ALTER TABLE 테이블명 MODIFY 칼럼명 새데이터타입
- 제약 조건 추가: ALTER TABLE 테이블명 ADD CONSTRAINT 제약조건명 제약조건타입 (칼럼명)
- 제약 조건 삭제: ALTER TABLE 테이블명 DROP CONSTRAINT 제약조건명
- 기본값 설정: ALTER TABLE 테이블명 MODIFY 칼럼명 데이터타입 DEFAULT 기본값

예시

- - 칼럼 추가

ALTER TABLE EMPLOYEE ADD PHONE_NUMBER VARCHAR2(15) NOT NULL;

- - 칼럼 삭제

ALTER TABLE EMPLOYEE DROP COLUMN PHONE_NUMBER;

- - 칼럼명 변경

ALTER TABLE EMPLOYEE RENAME COLUMN NAME TO FULL_NAME;

- - 데이터 타입 변경

ALTER TABLE EMPLOYEE MODIFY SALARY NUMBER(10, 2);

- - 제약 조건 추가

ALTER TABLE EMPLOYEE ADD CONSTRAINT CHK_SALARY CHECK (SALARY > 0);

- - 제약 조건 삭제

ALTER TABLE EMPLOYEE DROP CONSTRAINT CHK_SALARY;

- - 기본값 설정

ALTER TABLE EMPLOYEE MODIFY JOIN_DATE DATE DEFAULT SYSDATE;

### **3.3 RENAME TABLE**

테이블 이름을 변경하는 명령어입니다.

오라클 예시

RENAME EMP TO EMP_BACKUP;

SQL Server 예시

sp_rename 'EMP', 'EMP_BACKUP';

### **3.4 DROP TABLE**

DROP TABLE 명령어는 테이블을 삭제하는 명령어로, 테이블의 모든 데이터와 구조를 함께 제거합니다.

DROP TABLE EMP_BACKUP;

### **3.5 TRUNCATE TABLE**

TRUNCATE TABLE은 테이블의 모든 데이터를 빠르게 삭제하나 테이블 구조는 유지합니다.

TRUNCATE TABLE EMP_BACKUP;

DROP TABLE과 TRUNCATE TABLE의 차이점

| 특성 | DROP TABLE | TRUNCATE TABLE  
|  
| --- | --- | --- |  
| 구조 삭제 | 테이블 구조와 데이터 모두 삭제 | 데이터만 삭제, 구조 유지 |  
| 복구 가능성 | 일반적으로 복구 불가능 | 일반적으로 복구 불가능 |  
| 관련 객체 | 인덱스, 트리거, 제약 조건 등 모두 삭제 | 인덱스, 트리거, 제약 조건은 유지 |  
| 공간 할당 | 테이블 공간 반환 | 테이블 공간 초기화 |

## **4. DCL (데이터 제어어)**

DCL은 데이터베이스에 대한 접근 권한을 관리하는 명령어입니다. 주요 명령어로는 GRANT와 REVOKE가 있습니다.

### **4.1 사용자 계정과 권한**

데이터베이스 시스템에서는 사용자 계정을 생성하고 해당 계정에 필요한 권한을 부여합니다.

사용자 생성 예시 (Oracle)

CREATE USER username IDENTIFIED BY password;

### **4.2 권한의 종류**

- 시스템 권한: 데이터베이스 전체에 대한 권한 (예: CREATE TABLE, CREATE VIEW)
- 객체 권한: 특정 객체에 대한 권한 (예: SELECT, INSERT, UPDATE, DELETE)

### **4.3 GRANT**

GRANT 명령어는 사용자에게 권한을 부여합니다.

객체 권한 부여 형식

GRANT 권한1, 권한2, ... ON 객체명 TO 사용자명 [WITH GRANT OPTION];

WITH GRANT OPTION: 부여받은 권한을 다른 사용자에게 부여할 수 있는 권한까지 부여

예시

- - SELECT와 INSERT 권한 부여

GRANT SELECT, INSERT ON EMP TO bbigter;

- - 권한 전달 옵션 포함

GRANT SELECT ON EMP TO bbigter WITH GRANT OPTION;

### **4.4 REVOKE**

REVOKE 명령어는 사용자에게 부여한 권한을 회수합니다.

형식

REVOKE 권한1, 권한2, ... ON 객체명 FROM 사용자명;

예시

REVOKE SELECT, INSERT ON EMP FROM bbigter;

### **4.5 ROLE을 이용한 권한 관리**

ROLE은 여러 권한을 묶어 그룹화한 것으로, 권한 관리를 효율적으로 할 수 있습니다.

ROLE 생성 및 권한 부여

- - ROLE 생성

CREATE ROLE hr_role;

- - ROLE에 권한 부여

GRANT SELECT, INSERT, UPDATE ON EMP TO hr_role;

GRANT SELECT ON DEPT TO hr_role;

- - 사용자에게 ROLE 부여

GRANT hr_role TO user1, user2;

- - ROLE 삭제

DROP ROLE hr_role;

ROLE을 사용하면 여러 사용자에게 동일한 권한을 부여하고 관리하는 작업을 간소화할 수 있습니다.

### **정리 및 핵심 포인트**

1. DML은 행 단위 데이터 조작에 쓰이며, MERGE로 삽입·갱신을 한 번에 처리할 수 있다-대량 작업 시 서브쿼리·버퍼 캐싱을 활용하면 성능이 향상된다.
2. TCL은 트랜잭션의 일관성을 보장한다. COMMIT은 변경사항을 영구화하고, ROLLBACK은 트랜잭션 시작점 또는 SAVEPOINT까지 복원한다. ACID 원칙을 기억하라.
3. DDL은 객체 구조를 다룬다. TRUNCATE는 DELETE보다 빠르지만 롤백이 불가능하며, DROP은 구조·데이터·연관 객체를 모두 제거한다.
4. DCL은 접근 제어의 핵심이다. GRANT로 권한을 부여하고, REVOKE로 회수하며, ROLE로 여러 권한을 묶어 다수 사용자에게 일괄 적용한다.
5. 실무에서는 권한 최소화 원칙(Principle of Least Privilege)을 적용해 보안 사고를 예방하고, 트랜잭션 관리 규칙을 문서화해 데이터 무결성을 유지한다.

- **끝 -**
- **Before:** `[학생: 김철수, 과목: 자바, 파이썬]` (한 칸에 두 개!)
- **After (1NF):**
    - `[학생: 김철수, 과목: 자바]`
    - `[학생: 김철수, 과목: 파이썬]` (각 줄마다 값 1개씩!)
